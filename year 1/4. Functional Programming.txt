4. Functional Programming
บทที่ 1 แนวคิด Paradigm เชิงฟังก์ชัน (The Functional Programming Paradigm)
**การปะทะกันของสองวิธีคิด** //Imperative vs. Declarative:ในการเขียนโปรแกรมแบบดั้งเดิมที่เราคุ้นเคย หรือ Imperative Programming (เช่น C หรือ Java ยุคเก่า) เรามักจะเขียนโค้ดในลักษณะของ "คำสั่ง" (Commands) ที่บอกคอมพิวเตอร์ว่า "ต้องทำอย่างไร" (How) ทีละขั้นตอน เช่น การจองพื้นที่ตัวแปร การวนลูป และการเปลี่ยนสถานะ (State) ของตัวแปรไปเรื่อยๆ จนจบการทำงาน ในทางกลับกัน Functional Programming เป็นกิ่งก้านของแนวคิดแบบ Declarative ที่เปลี่ยนจากการบอกขั้นตอน (How) มาเป็นการบอกนิยามว่าผลลัพธ์ "คืออะไร" (What) โดยมีจุดเน้นดังนี้1 มองการคำนวณทุกอย่างเป็นการประเมินค่าฟังก์ชันทางคณิตศาสตร์ (Mathematical Expressions)2 ไม่มีการเปลี่ยนแปลงค่าตัวแปรเดิม (Immutability) แต่เป็นการส่งต่อผลลัพธ์จากฟังก์ชันหนึ่งไปยังอีกฟังก์ชันหนึ่งแทน //

**หัวใจสำคัญ การประเมินค่าฟังก์ชันทางคณิตศาสตร์**//ในกระบวนทัศน์เชิงฟังก์ชัน เราจะลดละความพยายามในการควบคุม "สถานะของตัวแปร" (Mutable State) และหันมาให้ความสำคัญกับ
1 Pure Functions ฟังก์ชันที่รับอินพุตเดิมจะได้ผลลัพธ์เดิมเสมอ และไม่มีผลข้างเคียง (Side Effects) เช่น การแอบไปเปลี่ยนค่าตัวแปรภายนอก หรือการเขียนไฟล์
2 Mathematical Thinking การมองปัญหาเหมือนสมการคณิตศาสตร์  f(x) = y  ที่มีความชัดเจนและทำนายผลลัพธ์ได้ 100% //

**ทำไมต้อง Functional Programming?**//เมื่อโปรแกรมมีขนาดใหญ่และซับซ้อนขึ้น ปัญหาที่ใหญ่ที่สุดที่นักพัฒนาต้องเจอคือ "สถานะที่คาดเดาไม่ได้" (Unexpected State Changes) ซึ่งเป็นต้นกำเนิดของบั๊กที่หายากที่สุด การเปลี่ยนมาใช้แนวคิดแบบ FP จึงให้ข้อดีมหาศาลดังนี้
1 Predictability โค้ดสามารถเดาผลลัพธ์ได้ง่าย เพราะไม่มี "พลังงานลึกลับ" จากภายนอกมาเปลี่ยนค่าตัวแปรในฟังก์ชัน
2 Readability & Conciseness โค้ดมักจะสั้นลงและอ่านเข้าใจง่ายขึ้น เพราะเน้นการบอก "เป้าหมาย" ของข้อมูลมากกว่า "วิธีการ" ประมวลผล
3 Concurrency Ready เนื่องจากไม่มีการเปลี่ยนสถานะตัวแปร (No Shared State) การเขียนโปรแกรมแบบทำงานพร้อมกัน (Parallelism/Concurrency) จึงทำได้ง่ายและปลอดภัยกว่าเดิมมาก โดยไม่ต้องกัง//

บทที่ 2 ฟังก์ชันในฐานะพลเมืองชั้นหนึ่ง (Functions as First-Class Citizens)
**พลังของพลเมืองชั้นหนึ่งทำอะไรได้บ้าง?**//เมื่อฟังก์ชันมีสถานะเป็นพลเมืองชั้นหนึ่ง นักพัฒนาจะสามารถจัดการกับฟังก์ชันในรูปแบบที่ยืดหยุ่นกว่าเดิมมาก ผ่าน 3 คุณสมบัติหลัก
1 การเก็บฟังก์ชันไว้ในตัวแปร (Assign to Variables) เราสามารถประกาศตัวแปรเพื่อ "เก็บ" ตรรกะของฟังก์ชันเอาไว้ได้ ทำให้เราสามารถเปลี่ยนพฤติกรรมของตัวแปรนั้นๆ ได้ตามบริบทที่ต้องการ
2 การส่งฟังก์ชันเป็นพารามิเตอร์ (Pass as Arguments) เราสามารถส่ง "วิธีคิด" หรือ "กฎเกณฑ์" เข้าไปในฟังก์ชันอื่นได้ ซึ่งเป็นจุดกำเนิดของโค้ดที่ยืดหยุ่นและนำกลับมาใช้ใหม่ได้สูง
3 การคืนค่าเป็นฟังก์ชัน (Return as Results) ฟังก์ชันสามารถสร้างและส่งฟังก์ชันใหม่ออกมาเป็นผลลัพธ์ได้ ทำให้เกิดเทคนิคขั้นสูงอย่างการสร้างฟังก์ชันเฉพาะทาง (Factory Functions) //

**ฟังก์ชันลำดับสูง (Higher-Order Functions: HOFs)**//เมื่อเราสามารถส่งหรือรับฟังก์ชันเป็นค่าได้ เราจะเข้าสู่ขอบเขตของ Higher-Order Functions (HOFs) ซึ่งเปรียบเสมือนฟังก์ชันที่อยู่เหนือฟังก์ชันอื่น มันไม่ได้ทำงานกับข้อมูลดิบเพียงอย่างเดียว แต่ทำงานกับ "ตรรกะ"1 ตัวอย่างที่เห็นได้ชัด ฟังก์ชันอย่าง map, filter หรือ reduce ที่เราใช้จัดการกับชุดข้อมูลขนาดใหญ่ ล้วนเป็น HOFs เพราะพวกมันรับ "กฎการแปลงข้อมูล" หรือ "เงื่อนไขการกรอง" ในรูปแบบของฟังก์ชันเข้าไปทำงานนั่นเอง //

**รากฐานจากแคลคูลัสแลมบ์ดา (Lambda Calculus)**//ความมหัศจรรย์นี้ไม่ได้เกิดขึ้นลอยๆ แต่มีรากฐานที่ลึกซึ้งมาจาก แคลคูลัสแลมบ์ดา (lambda-calculus) ซึ่งเป็นระบบคณิตศาสตร์ที่คิดค้นโดย Alonzo Church ในระบบนี้ ทุกอย่างคือฟังก์ชัน และฟังก์ชันคือทุกอย่าง1 แนวคิดเรื่อง ฟังก์ชันนิรนาม (Anonymous Functions) หรือที่ในภาษาสมัยใหม่มักเรียกว่า Lambda Functions หรือ Arrow Functions ก็ถือกำเนิดมาจากที่นี่2 มันคือฟังก์ชันที่ถูกสร้างขึ้นเพื่อใช้งานเฉพาะหน้าโดยไม่ต้องตั้งชื่อให้เสียเวลา ช่วยให้โค้ดกระชับ สะอาด และลดมลภาวะในการประกาศชื่อตัวแปรที่เกินความจำเป็น //

บทที่ 3 ความไม่เปลี่ยนแปลงและผลข้างเคียง (Immutability and Side Effects)
**ความไม่เปลี่ยนแปลง (Immutability)** //ข้อมูลที่แตะต้องไม่ได้:ในโปรแกรมทั่วไป เรามักจะเห็นการแก้ไขค่าในตัวแปรเดิม (Mutation) เช่น x = x + 1 แต่ใน FP เรายึดถือหลักการ Immutability คือเมื่อข้อมูลถูกสร้างขึ้นแล้ว มันจะคงอยู่อย่างนั้นไปตลอดวงจรชีวิตของมัน
1 No Mutation เราจะไม่ "แก้ไข" ข้อมูลเดิม แต่จะ "สร้างข้อมูลชุดใหม่" ขึ้นมาจากการดัดแปลงข้อมูลเดิมแทน
2 ข้อดี ช่วยกำจัดปัญหาเรื่องการแย่งกันเข้าถึงข้อมูล (Race Conditions) ในโปรแกรมที่ทำงานพร้อมกันหลายอย่าง เพราะไม่มีใครสามารถแอบไปเปลี่ยนค่าใต้จมูกของคนอื่นได้ ข้อมูลจึงมีความน่าเชื่อถือเสมอ //

**ฟังก์ชันบริสุทธิ์ (Pure Functions)** //กฎเหล็กแห่งความแม่นยำ:ฟังก์ชันใน FP จะต้องทำตัวเหมือนฟังก์ชันทางคณิตศาสตร์จริงๆ (f(x) = y) โดยต้องรักษาคุณสมบัติ 2 ข้ออย่างเคร่งครัด
1 Deterministic (Input เดิม → Output เดิม) ไม่ว่าคุณจะรันฟังก์ชันนี้กี่ล้านครั้ง หรือรันที่ไหนบนโลก ถ้าใส่ค่า A เข้าไป ผลลัพธ์ต้องออกมาเป็น B เสมอ ไม่ขึ้นอยู่กับเวลา ไม่ขึ้นอยู่กับตัวแปรสุ่ม หรือปัจจัยภายนอกใดๆ
2 No Side Effects (ไร้ผลข้างเคียง) ฟังก์ชันมีหน้าที่ "คำนวณและคืนค่า" เท่านั้น มันจะต้องไม่แอบไปทำกิจกรรมอื่นๆ ที่ส่งผลต่อโลกภายนอก เช่น ไม่ไปเปลี่ยนค่าตัวแปร Global ไม่เขียนไฟล์ลงฮาร์ดดิสก์ ไม่พิมพ์ข้อความออกหน้าจอ หรือไม่อัปเดตข้อมูลลงฐานข้อมูล //

**พลังของการคาดเดาได้ (Predictability)** //เมื่อคุณรวบเอา Immutability และ Pure Functions เข้าด้วยกัน สิ่งที่ได้คือโค้ดที่ "คาดเดาผลลัพธ์ได้อย่างแม่นยำ 100%"
1 การทดสอบที่ง่ายขึ้น (Easy Testing) การเขียน Unit Test จะกลายเป็นเรื่องง่ายมาก เพราะคุณไม่ต้องเตรียม "สถานะเริ่มต้น" (Setup State) ที่ยุ่งยาก ไม่ต้องล้างข้อมูลหลังทดสอบ (Teardown)
2 คุณแค่ส่งค่า Input เข้าไปและตรวจสอบว่า Output ตรงกับที่คิดไว้หรือไม่ ฟังก์ชันจะแยกตัวเป็นอิสระจากส่วนอื่นๆ ของโปรแกรมอย่างสมบูรณ์ //

บทที่ 4 การประมวลผลรายการข้อมูล (Processing Collections)
**Map การแปลงร่างข้อมูล (The Transformer)** //Map คือการนำฟังก์ชันหนึ่งไปประยุกต์ใช้กับสมาชิก ทุกตัว ในรายการ เพื่อสร้างรายการใหม่ที่มีจำนวนสมาชิกเท่าเดิม แต่เปลี่ยนรูปร่างไปตามที่เราต้องการ นิยามทางคณิตศาสตร์คือ หากเรามีเซต A และฟังก์ชัน f: A \to B การ Map จะทำให้เราได้เซต B ที่เป็นผลลัพธ์จากการประยุกต์ใช้ f กับสมาชิกทุกตัวใน A1 ตัวอย่าง การเปลี่ยนรายการราคาสินค้า (ตัวเลข) ให้เป็นข้อความที่มีสัญลักษณ์สกุลเงินกำกับ หรือการดึงเฉพาะชื่อออกมาจาก Object ของผู้ใช้งาน //

**Filter การคัดกรอง (The Gatekeeper)**//Filter ทำหน้าที่เป็นประตูกลที่จะคัดเลือกสมาชิกไว้เฉพาะตัวที่ผ่านเงื่อนไข (Predicate) ที่เรากำหนดเท่านั้น โดยผลลัพธ์ที่ได้จะเป็นรายการใหม่ที่มีจำนวนสมาชิกเท่าเดิมหรือน้อยลง หลักการคือมันจะรับฟังก์ชันที่คืนค่าเป็น Boolean (True/False)
1 หากตัวไหนเป็น True จะได้ไปต่อ หากเป็น False จะถูกคัดออก
2 ตัวอย่าง การคัดกรองเฉพาะ "เลเวลที่สูงกว่า 50" ในเกม หรือการเลือกเฉพาะ "บทความที่มีสถานะ Published" ออกมาจากฐานข้อมูล //

**Reduce การรวบยอด (The Accumulator/Folder)** //Reduce (หรือบางภาษาเรียกว่า Fold) คือฟังก์ชันที่ทรงพลังที่สุดและซับซ้อนที่สุดในบรรดาสามตัวนี้ หน้าที่ของมันคือการนำสมาชิกทั้งหมดในรายการมา "ยุบรวม" ให้เหลือเพียงผลลัพธ์เดียว การทำงานคือมันจะใช้ตัวสะสม (Accumulator) และค่าเริ่มต้น (Initial Value) จากนั้นจะค่อยๆ นำสมาชิกแต่ละตัวในรายการมาประมวลผลร่วมกับตัวสะสมไปเรื่อยๆ จนครบ1 ตัวอย่าง การหาผลรวมคะแนนสอบทั้งหมด (Sum), การหาค่าสูงสุด (Max), หรือแม้แต่การเปลี่ยน Array ของตัวอักษรให้กลายเป็นประโยคเพียงประโยคเดียว //

**พลังของการต่อท่อ (Functional Pipelines)** //จุดเด่นที่สุดของการใช้ Map, Filter และ Reduce คือการที่เราสามารถ "ต่อท่อ" (Chaining) พวกมันเข้าด้วยกันได้ ทำให้โค้ดที่ซับซ้อนดูเรียบง่ายเหมือนการอ่านประโยคภาษาอังกฤษ โจทย์เช่น ค้นหารายชื่อนักเรียนที่สอบผ่าน (คะแนน > 50) และเพิ่มโบนัสให้คนละ 5 คะแนน จากนั้นหาคะแนนรวมทั้งหมด1 Filter เลือกเฉพาะคนที่คะแนน > 502 Map นำคะแนนแต่ละคนมา +53 Reduce นำผลลัพธ์ทั้งหมดมาบวกกันเป็นก้อนเดียว //

บทที่ 5 การเรียกซ้ำและการพิสูจน์ความถูกต้อง (Recursion and Tail-Call Optimization)
**ปัญหาระดับโครงสร้าง** //Call Stack และ Stack Overflow:ในการเรียกซ้ำแบบทั่วไป (General Recursion) ทุกครั้งที่ฟังก์ชันเรียกตัวเอง คอมพิวเตอร์จะต้องจองพื้นที่ในหน่วยความจำที่เรียกว่า "Stack Frame" เพื่อจดจำสถานะและตัวแปรท้องถิ่นของฟังก์ชันเดิมไว้ เผื่อว่าเมื่อฟังก์ชันลูกทำงานเสร็จ จะได้กลับมาทำงานต่อที่ฟังก์ชันแม่ได้ถูกต้อง
1 วิกฤตหน่วยความจำ หากเราต้องทำซ้ำข้อมูลนับแสนรายการ การเรียกซ้ำจะสร้าง Stack Frame ซ้อนกันนับแสนชั้น จนเกินขีดจำกัดของหน่วยความจำที่ระบบจัดสรรไว้ ส่งผลให้เกิดข้อผิดพลาดรุนแรงที่เรียกว่า "Stack Overflow" //

**เทคนิคการแก้ปัญหา Tail-Call Optimization (TCO)**//เพื่อทลายขีดจำกัดเรื่องหน่วยความจำ ภาษาที่รองรับ FP อย่างเต็มตัว (เช่น Haskell, Scala, หรือ Scheme) รวมถึงคอมไพเลอร์สมัยใหม่ จะมีฟีเจอร์ที่เรียกว่า Tail-Call Optimization (TCO) เข้ามาช่วยจัดการ
1 หัวใจของ TCO คือการเขียน "Tail Recursion" ฟังก์ชันจะถูกเรียกว่าเป็น Tail Recursive ก็ต่อเมื่อ "การเรียกตัวเองซ้ำ เป็นคำสั่งสุดท้ายของฟังก์ชันนั้นจริงๆ" โดยไม่มีการรอเอาผลลัพธ์ไปประมวลผลต่อ
2 กลไกการทำงาน เมื่อคอมไพเลอร์ตรวจพบว่าเป็นการเรียกแบบ Tail Call มันจะ "ทำลาย" Stack Frame เดิมทิ้งแล้ว "แทนที่" ด้วยเฟรมใหม่ทันที หรือเป็นการวนกลับไปเริ่มทำงานที่หัวฟังก์ชันใหม่โดยใช้พื้นที่หน่วยความจำเดิม
3 ผลลัพธ์มหาศาล การเรียกซ้ำด้วย TCO จะใช้หน่วยความจำคงที่ หรือ O(1) ซึ่งมีประสิทธิภาพเทียบเท่ากับการวนลูปปกติ (Iteration) //

**การออกแบบลอจิกด้วยตัวสะสม (Accumulators)** //การจะเปลี่ยนฟังก์ชันแบบปกติให้กลายเป็น Tail Recursion มักต้องใช้เทคนิค "ตัวสะสม" (Accumulators) ซึ่งเป็นการส่ง "ผลลัพธ์ชั่วคราว" ไปกับพารามิเตอร์ของฟังก์ชันในรอบถัดไป แทนที่จะเก็บไว้รอประมวลผลใน Stack Frame เดิม1 Recursion ทั่วไป return n + factorial(n-1) (ต้องรอผลลัพธ์จาก factorial มาบวก n จึงไม่ใช่ Tail Call)2 Tail Recursion return factorial(n-1, n * accumulator) (ส่งผลคูณสะสมไปให้รอบหน้าทันที ไม่ต้องรอใคร) //

**การรับประกันความถูกต้อง (Correctness Proof)** //ข้อดีแฝงของการใช้ Recursion คือความใกล้เคียงกับนิยามทางคณิตศาสตร์ ทำให้เราสามารถใช้ "การอุปนัยเชิงคณิตศาสตร์" (Mathematical Induction) มาพิสูจน์ความถูกต้องของโค้ดได้ง่ายขึ้น1 Base Case พิสูจน์ว่าโปรแกรมทำงานถูกที่จุดหยุด2 Inductive Step พิสูจน์ว่าหากขั้นตอนก่อนหน้าถูก ขั้นตอนถัดไปต้องถูกเสมอ //

บทที่ 6 การจับคู่รูปแบบและชนิดข้อมูล (Pattern Matching and Algebraic Data Types)
**ชนิดข้อมูลพีชคณิต (Algebraic Data Types - ADTs)**//ADTs คือวิธีคิดในการออกแบบ "รูปร่าง" ของข้อมูลที่รัดกุมและสื่อความหมายได้ลึกซึ้งกว่าชนิดข้อมูลพื้นฐานทั่วไป โดยแบ่งออกเป็น 2 รูปแบบหลักที่ทำงานร่วมกัน
1 Sum Types (หรือ Union Types) เป็นการประกาศว่าข้อมูลนี้จะเป็น "อย่างใดอย่างหนึ่ง" จากกลุ่มที่กำหนดไว้ เช่น ข้อมูลการส่งเงินอาจเป็นได้แค่ Success ที่เก็บยอดเงิน หรือ Failure ที่เก็บข้อความข้อผิดพลาดเท่านั้น การใช้ Sum Types ช่วยกำจัดสถานะที่เป็นไปไม่ได้ (Illegal States) ออกจากระบบของคุณ
2 Product Types คือการรวบรวมข้อมูลหลายประเภทเข้าไว้ด้วยกัน (คล้ายกับ Struct หรือ Object) เช่น Student ที่ประกอบด้วย Name และ ID //

**การจับคู่รูปแบบ (Pattern Matching)** //มากกว่าแค่การเช็คเงื่อนไข:ถ้า ADTs คือการสร้างกล่องข้อมูลที่ชัดเจน Pattern Matching ก็คือเครื่องมือที่ใช้ "เปิดกล่อง" นั้นอย่างชาญฉลาด มันทำงานได้เหนือกว่า switch-case ทั่วไปใน 2 มิติหลัก
1 โครงสร้างและการดึงข้อมูล (Deconstruction & Extraction) มันสามารถตรวจสอบลึกลงไปในโครงสร้างของข้อมูลและ "ดึง" ตัวแปรข้างในออกมาใช้งานได้ทันทีในบรรทัดเดียวกัน เช่น ถ้าจับคู่เจอรูปแบบ Success(amount) เราจะสามารถนำตัวแปร amount ไปคำนวณต่อได้ทันที
2 ความรัดกุมทางตรรกะ มันสามารถตรวจสอบเงื่อนไขที่ซับซ้อน (Guards) ไปพร้อมๆ กับการตรวจสอบชนิดข้อมูลได้เลย ทำให้โค้ดที่เคยซ้อนกันหลายชั้น (Nested If) ยุบรวมเหลือเพียงรายการของรูปแบบที่อ่านง่ายและชัดเจน //

**การตรวจสอบที่ครอบคลุม (Exhaustive Checking)**//นี่คือฟีเจอร์ที่เป็นเสมือน "ตาข่ายนิรภัย" สำหรับนักพัฒนา เมื่อคุณใช้ Pattern Matching ร่วมกับ ADTs คอมไพเลอร์จะเข้ามาช่วยคุณตรวจสอบว่า "คุณเขียนดักครบทุกกรณีแล้วหรือยัง?"
1 ตัวอย่าง หากคุณนิยามผลลัพธ์ไว้ 3 แบบ แต่คุณเขียนโค้ดดักไว้เพียง 2 แบบ คอมไพเลอร์จะแจ้งเตือนทันทีว่าคุณลืมกรณีที่ 3 ไป
2 คุณลักษณะนี้ช่วยกำจัดบั๊กประเภท "Runtime Error" หรือเคสที่ตกหล่นไป (Edge Cases) ซึ่งมักเป็นสาเหตุหลักที่ทำให้แอปพลิเคชันล่มในเวลาที่คาดไม่ถึง //

**ประโยชน์ในเชิงวิศวกรรมซอฟต์แวร์**//การเปลี่ยนจาก if-else มาเป็น ADTs และ Pattern Matching จะยกระดับโค้ดของคุณในหลายด้าน
1 Safety ระบบจะมีความปลอดภัยมากขึ้น เพราะข้อมูลถูกบังคับให้อยู่ในรูปแบบที่ถูกต้องเสมอ
2 Maintainability เมื่อคุณต้องการเพิ่มกรณีใหม่ (New Case) คอมไพเลอร์จะชี้จุดที่คุณต้องแก้ไขให้ทั่วทั้งโปรแกรมโดยอัตโนมัติ
3 Clarity โค้ดจะอ่านเหมือนการบรรยายกฎเกณฑ์ทางธุรกิจ (Business Rules) มากกว่าคำสั่งคอมพิวเตอร์ ทำให้การสื่อสารในทีมทำได้ง่ายขึ้น //

บทที่ 7 การประมวลผลแบบเชื่องช้า (Lazy Evaluation)
**แนวคิดหลักคำนวณเมื่อจำเป็น (Call-by-Need)** //Lazy Evaluation คือกลยุทธ์การประมวลผลที่คอมพิวเตอร์จะไม่ลงมือคำนวณค่าใดๆ จนกว่าจะถึงวินาทีที่ค่านั้น "จำเป็นต้องถูกใช้งานจริง" เท่านั้น ระหว่างที่ยังไม่ใช้ คอมพิวเตอร์จะเก็บสิ่งที่เรียกว่า "Thunk" (หรือบันทึกคำสั่งและบริบทที่ใช้คำนวณ) เอาไว้แทนผลลัพธ์จริง1 Eager (ใจร้อน) สั่งบวกเลข 1 + 1 ปุ๊บ ได้ 2 ทันทีและจองที่ใน RAM ปั๊บ2 Lazy (ใจเย็น) สั่งบวกเลข 1 + 1 ปุ๊บ คอมพิวเตอร์จะจดบันทึกไว้ว่า "มีงานต้องบวกเลขนะ" แต่จะยังไม่บวก จนกว่าจะมีฟังก์ชันอื่นมาขอดูผลลัพธ์เพื่อนำไปแสดงผลหรือคำนวณต่อ //

**ความมหัศจรรย์ของลิสต์ที่ยาวอนันต์ (Infinite Data Structures)** //หนึ่งในประโยชน์ที่ฟังดูเหมือนเวทมนตร์ของ Lazy Evaluation คือการที่เราสามารถจัดการกับ "ข้อมูลที่ไม่มีจุดสิ้นสุด" (Infinite Lists) ได้โดยที่หน่วยความจำไม่เต็ม ตัวอย่างเช่น1 ลำดับเลขจำนวนเต็มทั้งหมด คุณสามารถประกาศตัวแปร allIntegers = [1..] ซึ่งเก็บเลขตั้งแต่ 1 ไปจนถึงอินฟินิตี้ได้ เพราะคอมพิวเตอร์จะไม่ได้สร้างเลขทั้งหมดขึ้นมาจริงๆ แต่มันเตรียมตัวที่จะ "สร้างเลขตัวถัดไป" เมื่อคุณเรียกใช้เท่านั้น2 ลำดับฟีโบนัชชี เราสามารถนิยามลำดับฟีโบนัชชีทั้งจักรวาลไว้ในตัวแปรเดียว แล้วสั่งว่า "ขอแค่ 10 ตัวแรกมาแสดงผล" ระบบก็จะคำนวณแค่ 10 รอบแล้วหยุดทำงานทันที3 ประสิทธิภาพเชิงหน่วยความจำ หากเราต้องการหาเลขตัวแรกในลิสต์ที่ผ่านเงื่อนไขบางอย่างจากข้อมูลหมื่นล้านเรกคอร์ด Lazy Evaluation จะค่อยๆ ดึงมาทีละตัว ประมวลผลทีละนิด และหยุดทันทีเมื่อเจอเป้าหมาย ทำให้ใช้ RAM เพียงเล็กน้อย (O(1) space) //

**การรีดประสิทธิภาพด้วยความ "เชื่องช้า" (Optimization)**//Lazy Evaluation ไม่ได้มีไว้เพื่อความเท่ทางคณิตศาสตร์เท่านั้น แต่มันคือเครื่องมือในการทำ Optimization ระดับสูง
1 หลีกเลี่ยงการประมวลผลที่ไม่จำเป็น (Short-circuiting) หากคุณมีฟังก์ชันที่ซับซ้อนมากสองตัวเชื่อมกันด้วยเงื่อนไข ถ้าผลลัพธ์จากตัวแรกเพียงพอที่จะตัดสินใจได้แล้ว ตัวที่สองจะไม่ถูกรันเลยแม้แต่บรรทัดเดียว
2 การรวมท่อประมวลผล (Pipeline Fusion) แทนที่คอมพิวเตอร์จะวนลูปสร้างอาร์เรย์ใหม่หลายรอบ (Intermediate Arrays) Lazy Evaluation จะรวมคำสั่งเหล่านั้นแล้ววนลูปเพียงรอบเดียวเพื่อคำนวณผลลัพธ์สุดท้าย
3 การจัดการ Streams เหมาะอย่างยิ่งกับข้อมูลเรียลไทม์ที่ไหลมาไม่หยุด เพราะเราสามารถนิยามวิธีการจัดการข้อมูลไว้ล่วงหน้า แล้วให้ระบบลงมือทำเฉพาะเมื่อมีข้อมูลไหลเข้ามาจริงๆ //

**ข้อควรระวัง** //ด้านมืดของความเชื่องช้า:แม้จะดูสมบูรณ์แบบ แต่ความเชื่องช้าก็มีราคาที่ต้องจ่าย
1 Memory Leaks (Space Leaks) หากเราเก็บ Thunks ไว้มากเกินไปโดยไม่ยอมใช้งานสักที Thunks เหล่านั้นอาจจะพอกพูนจนกินพื้นที่หน่วยความจำมากกว่าผลลัพธ์จริงๆ
2 Debugging ที่ยากขึ้น เนื่องจากโค้ดไม่ได้รันตามลำดับบรรทัด แต่รันตามความต้องการใช้ค่า การไล่หาบั๊กจึงมีความซับซ้อนกว่าปกติ //

บทที่ 8 พื้นฐาน Functors และ Monads (Foundations of Functors and Monads)
**Functor กล่องที่อนุญาตให้เรา 'Map' ข้อมูลข้างใน** //ก่อนจะเข้าใจ Monad เราต้องผ่านด่านของ Functor เสียก่อน ลองจินตนาการว่า Functor คือ "กล่อง" (Container) ที่ห่อหุ้มข้อมูลบางอย่างเอาไว้ และมันมีคุณสมบัติพิเศษคือเราสามารถใช้ฟังก์ชัน map เพื่อเปลี่ยนแปลงข้อมูลที่อยู่ข้างในได้ โดยที่ไม่ต้อง "แกะ" กล่องออกมาด้วยตัวเอง1 นิยาม หากเรามีกล่องที่บรรจุค่า a และเรามีฟังก์ชัน f: a \to b การ map จะทำให้เราได้กล่องที่บรรจุค่า b แทน2 ทำไมต้องห่อ? การห่อข้อมูลช่วยให้เราสามารถประมวลผลข้อมูลภายใต้เงื่อนไขบางอย่างได้ เช่น ข้อมูลนั้นอาจจะ "มีค่า" หรือ "ไม่มีค่า" (Maybe) หรืออาจจะเป็นข้อมูลที่ต้องรอจากเครือข่าย (Promise) //

**Monad การต่อท่อ (Pipeline) สำหรับฟังก์ชันที่คืนค่าเป็นกล่อง** //ความซับซ้อนจะเกิดขึ้นเมื่อฟังก์ชันที่เราต้องการใช้ ดันส่งผลลัพธ์ออกมาเป็น "กล่อง" เช่นกัน หากเราใช้ map ปกติ เราจะได้กล่องที่ซ้อนอยู่ในกล่องอีกที (เช่น Maybe(Maybe(Value))) ซึ่งจัดการยากมาก Monad จึงเข้ามาแก้ปัญหานี้ด้วยปฏิบัติการที่เรียกว่า flatMap (หรือ bind)
1 หัวใจของ Monad คือการอนุญาตให้เรานำฟังก์ชันที่ส่งคืน Context (เช่น การจัดการ Error หรือ Side Effect) มาเรียงต่อกันเป็นสาย
2 กลไก หน้าที่ของมันคือการ "คลี่" (Flatten) กล่องที่ซ้อนกันให้เหลือเพียงชั้นเดียว พร้อมทั้งเชื่อมต่อฟังก์ชันเข้าด้วยกันเป็นทอดๆ โดยที่ Context เหล่านั้นจะถูกส่งต่อและจัดการให้อัตโนมัติเบื้องหลัง //

**การจัดการความล้มเหลวด้วย Maybe/Option Monad** //ตัวอย่างที่คลาสสิกที่สุดคือการจัดการกับค่าว่างหรือ null ในภาษาทั่วไปเราต้องเขียน if (x != null) ซ้อนกันหลายชั้น แต่ใน FP เราจะใช้ Maybe Monad1 Just(Value) มีข้อมูลอยู่ข้างใน2 Nothing ไม่มีข้อมูลเมื่อเราใช้ Monad ในการต่อท่อฟังก์ชัน หากมีขั้นตอนใดขั้นตอนหนึ่งส่งคืนค่า Nothing ออกมา ทุกขั้นตอนหลังจากนั้นจะถูกข้ามไปโดยอัตโนมัติ และผลลัพธ์สุดท้ายจะเป็น Nothing อย่างปลอดภัย โดยที่คุณไม่ต้องเขียน if เช็คค่าว่างแม้แต่บรรทัดเดียว //

**จุดสูงสุดของการเขียนโปรแกรมที่บริสุทธิ์ (Pure Context)** //นอกจากเรื่องค่าว่างแล้ว Monad ยังถูกนำไปใช้จัดการสิ่งที่เป็น "มลพิษ" ต่อ Pure Functions นั่นคือ Side Effects
1 IO Monad จัดการการอ่านเขียนไฟล์หรือรับส่งข้อมูลผ่านระบบเครือข่าย
2 State Monad จัดการกับตัวแปรที่ต้องมีการเปลี่ยนแปลงสถานะ
3 Result/Either Monad จัดการกับ Error ที่มาพร้อมกับข้อความอธิบายเหตุผล //

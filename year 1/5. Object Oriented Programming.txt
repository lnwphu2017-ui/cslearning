5. Object Oriented Programming
บทที่ 1 พื้นฐานวัตถุและคลาส (Basics of Objects and Classes)
**คลาส (Class) vs วัตถุ (Object)** //พิมพ์เขียวกับผลผลิต:เพื่อให้เห็นภาพชัดเจนที่สุด เรามักจะเปรียบเทียบกับวงการก่อสร้างหรืออุตสาหกรรมครับ
1 คลาส (Class) เปรียบเสมือน "พิมพ์เขียว" (Blueprint) หรือแม่แบบแม่พิมพ์ มันไม่ใช่บ้านที่เข้าไปอยู่ได้จริงๆ แต่เป็นแบบร่างที่ระบุว่าบ้านหลังนี้ "ควรมี" อะไรบ้าง (เช่น มีกี่ห้องนอน มีสีอะไร)
2 วัตถุ (Object) คือ "บ้านที่สร้างเสร็จแล้ว" ซึ่งถูกสร้างขึ้นตามพิมพ์เขียวใบนั้น เราสามารถสร้างบ้านกี่หลังก็ได้จากพิมพ์เขียวใบเดียว โดยบ้านแต่ละหลังอาจจะมีสีทาบ้านต่างกัน แต่โครงสร้างพื้นฐานยังเหมือนเดิมตามแบบร่าง
3 การสร้างอินสแตนซ์ (Instantiation) คือกระบวนการที่เราเปลี่ยนจาก "แบบร่างในกระดาษ" (Class) ให้กลายเป็น "วัตถุที่มีตัวตนในหน่วยความจำ" (Object) ครับ //

**องค์ประกอบภายในวัตถุ สถานะและพฤติกรรม** //ในฐานะนักพัฒนา เราจะนิยามวัตถุผ่าน 2 ส่วนหลักที่ทำงานร่วมกัน
1 สถานะ (State) ผ่าน Attributes/Fields คือข้อมูลหรือลักษณะเฉพาะที่วัตถุนั้นถือครองอยู่ (มักจะเป็นคำนาม) เช่น หากคลาสคือ Car สถานะอาจจะเป็น color, model, currentSpeed
2 พฤติกรรม (Behavior) ผ่าน Methods คือความสามารถหรือการกระทำที่วัตถุนั้นทำได้ (มักจะเป็นคำกริยา) เช่น วัตถุรถยนต์สามารถ accelerate(), brake(), หรือ turnKey()
การรวมเอาข้อมูล (State) และฟังก์ชัน (Behavior) มาไว้ในก้อนเดียวกันแบบนี้ คือหนึ่งในหลักการเบื้องต้นของ OOP ที่ช่วยป้องกันไม่ให้โค้ดส่วนอื่นแอบมาแก้ไขข้อมูลของเราโดยตรงโดยไม่ผ่านพฤติกรรมที่เรากำหนดไว้ //

**ทำไมต้อง OOP?** //เมื่อโปรแกรมขยายใหญ่ขึ้น การเขียนโค้ดแบบเรียงลำดับลงมาเรื่อยๆ (Procedural) จะทำให้การจัดการความสัมพันธ์ของข้อมูลทำได้ยากมาก การใช้ OOP จะเปลี่ยนวิธีคิดให้มองว่า
1 แทนที่จะมองว่า "มีฟังก์ชันเช็คคะแนน" และ "มีตัวแปรเก็บคะแนน" แยกกัน
2 ให้มองว่า "มีวัตถุ Student" ที่รู้คะแนนของตัวเอง และมีพฤติกรรมในการ "แสดงเกรด" ของตัวเองได้ //

บทที่ 2 การห่อหุ้มและการสิทธิ์เข้าถึง (Encapsulation and Access Control)
**การซ่อนข้อมูล (Data Hiding)** //จินตนาการว่าคุณกำลังขับรถ (Object Car) คุณมีหน้าที่เพียงเหยียบคันเร่งหรือหมุนพวงมาลัย คุณไม่จำเป็นต้อง (และไม่ควร) เข้าไปยุ่งกับจังหวะการฉีดน้ำมันในเครื่องยนต์โดยตรง เพราะหากคุณปรับจังหวะผิดเพียงนิดเดียว เครื่องยนต์อาจพังได้ ในทางโปรแกรมมิ่งก็เช่นกัน เราซ่อนตัวแปร (State) ไว้เพื่อ
ป้องกันความเสียหาย ไม่ให้โค้ดส่วนอื่นมาแก้ค่าจนระบบรวน (เช่น การแก้ค่า "ยอดเงินในบัญชี" เป็นติดลบ)
ลดความซับซ้อน ผู้ใช้งานคลาสของคุณแค่เรียกใช้ฟังก์ชันที่เตรียมไว้ให้ โดยไม่ต้องรู้ว่าข้างในทำงานยังไง (Abstraction) //

**ตัวกำหนดสิทธิ์เข้าถึง (Access Modifiers)** //ยามเฝ้าประตูเราใช้ Access Modifiers เป็นตัวระบุว่าใครมีสิทธิ์มองเห็นหรือใช้งานสมาชิกในคลาสบ้าง โดยมี 2 ตัวหลักที่ต้องใช้ให้คล่องคือ
private (ส่วนตัว) ข้อมูลจะถูกเห็นและใช้งานได้ เฉพาะภายในคลาสตัวเองเท่านั้น เปรียบเหมือนความลับในบ้านที่คนนอกห้ามรู้ นี่คือหัวใจของการทำ Data Hiding
public (สาธารณะ) เปิดให้ทุกคลาสในโปรแกรมเข้าถึงได้ มักใช้กับ Method ที่ต้องการให้คนอื่นเรียกใช้งาน (Interface)
นอกจากนี้ยังมี protected ที่เกี่ยวข้องกับการสืบทอด ซึ่งเราจะคุยกันในบทถัดไปครับ //

**Getters & Setters ยามรักษาการณ์ (The Guards)**
//เมื่อเราล็อคประตูด้วย private แล้ว เราจะเปิดช่องทางเล็กๆ ผ่าน Method พิเศษ เพื่อให้โลกภายนอกยังสามารถปฏิสัมพันธ์กับข้อมูลได้แบบ "มีเงื่อนไข"
Getter (การอ่าน) ทำหน้าที่ส่งค่ากลับไปให้ผู้เรียก แทนที่จะให้เขามาหยิบเอง เราเป็นคนยื่นให้ (อ่านได้อย่างเดียว แก้ไขไม่ได้)
Setter (การเขียน) ทำหน้าที่รับค่ามาอัปเดตตัวแปร แต่ที่สำคัญที่สุดคือเราสามารถใส่ Logic ตรวจสอบ (Validation) ได้
ตัวอย่าง ถ้ามีคนสั่ง setAge(-5) ยามรักษาการณ์ (Setter) จะตรวจสอบทันทีว่า "อายุติดลบไม่ได้นะ!" และปฏิเสธการอัปเดตค่าทันที เพื่อรักษาความถูกต้องของข้อมูลในระบบ//

บทที่ 3 คอนสตรัคเตอร์และการจัดการทรัพยากร (Constructors and Resource Management)

**คอนสตรัคเตอร์ (Constructor) พิธีแรกรับของวัตถุ**//คอนสตรัคเตอร์ คือเมธอดพิเศษที่มีชื่อเดียวกับคลาส และไม่มีการส่งค่ากลับ (Return Type) มันจะถูกเรียกใช้งานเพียง "ครั้งเดียว" ในวินาทีที่วัตถุถูกสร้างขึ้น
1 Default Constructor (ไม่มีพารามิเตอร์) ใช้สำหรับกำหนดค่าเริ่มต้นพื้นฐานให้กับทุกวัตถุเหมือนๆ กัน
2 Parameterized Constructor (มีพารามิเตอร์) ช่วยให้เราสามารถส่งค่าเฉพาะเจาะจงเข้าไปให้วัตถุตั้งแต่เกิดได้เลย เช่น สร้างวัตถุ User พร้อมกำหนด username ทันทีโดยไม่ต้องรอเรียก Setter ในภายหลัง
Pro Tip การใช้ Constructor ช่วยรับประกันว่า "วัตถุจะไม่ตกอยู่ในสถานะที่ไม่สมบูรณ์" เช่น การมีวัตถุ Student ที่ไม่มีชื่อหรือรหัสประจำตัว เพราะเราสามารถบังคับให้ต้องส่งข้อมูลเหล่านี้ผ่าน Constructor ได้เท่านั้น //

**วงจรชีวิตและการทำลายวัตถุ (Destruction)**
//ในภาษาโปรแกรมยุคเก่าอย่าง C++ นักพัฒนาต้องรับผิดชอบ "การทำลาย" วัตถุด้วยตัวเองผ่าน Destructor เพื่อคืนหน่วยความจำให้ระบบ หากลืมทำลายจะเกิดปัญหา Memory Leak แต่ในภาษาสมัยใหม่ส่วนใหญ่จะมีระบบที่เรียกว่า
Garbage Collection (GC)
เปรียบเสมือน "พนักงานทำความสะอาดเทศบาล" ที่คอยวิ่งตรวจเช็คใน RAM ตลอดเวลาว่า "วัตถุชิ้นไหนไม่มีใครอ้างอิงถึง (Reference) แล้วบ้าง?" หากพบว่าวัตถุตัวนั้นถูกทิ้งไว้กลางทางและไม่มีโค้ดส่วนไหนเรียกใช้อีก GC จะทำการลบวัตถุนั้นทิ้งและคืนพื้นที่หน่วยความจำให้โดยอัตโนมัติ//

**ปัญหาหน่วยความจำรั่วไหล (Memory Leak)**//แม้จะมี Garbage Collection แต่ Memory Leak ก็ยังเกิดขึ้นได้หากนักพัฒนา "ถือสายจูง" วัตถุไว้ไม่ยอมปล่อย เช่น การเก็บวัตถุไว้ในลิสต์ขนาดใหญ่ที่เป็นโกลบอล (Global State) ตลอดเวลา แม้จะไม่ได้ใช้งานแล้วก็ตาม
ผลกระทบ โปรแกรมจะค่อยๆ กิน RAM มากขึ้นเรื่อยๆ จนเครื่องเริ่มอืด (Lag) และในที่สุดโปรแกรมก็จะค้างหรือ "Crash" ไปเนื่องจากหน่วยความจำเต็ม
การจัดการ การออกแบบโครงสร้างความสัมพันธ์ของวัตถุให้ชัดเจน และการรู้ว่าเมื่อไหร่ควรปล่อยการอ้างอิง (เช่น การ Set ค่าให้เป็น null) คือทักษะสำคัญที่ทำให้โปรแกรมของคุณทำงานได้ลื่นไหลในระยะยาว //

บทที่ 4 การสืบทอด (Inheritance)
**ความสัมพันธ์แบบ "IS-A" (ความเป็นประเภทหนึ่ง)**//หัวใจสำคัญของการสืบทอดคือการมองหาความสัมพันธ์ที่ว่า "สิ่งนี้... เป็น... สิ่งนั้นประเภทหนึ่ง"
รถสปอร์ต เป็น (IS-A) รถยนต์
พนักงานรายเดือน เป็น (IS-A) พนักงาน
สุนัข เป็น (IS-A) สัตว์เลี้ยง
เมื่อเรากำหนดความสัมพันธ์นี้ได้แล้ว คลาสลูก (Subclass) จะสามารถเข้าถึงความสามารถ (Methods) และข้อมูล (Attributes) ของ คลาสแม่ (Superclass) ได้ทันทีโดยไม่ต้องเขียนใหม่แม้แต่บรรทัดเดียว //

**คลาสแม่ (Superclass) vs คลาสลูก (Subclass)**
//Superclass (คลาสแม่/คลาสหลัก)เป็นคลาสที่เก็บลักษณะ "ร่วม" ที่ทุกอย่างในกลุ่มนี้ต้องมี เช่น รถยนต์ทุกคันต้องมี brand และมีฟังก์ชัน drive()
Subclass (คลาสลูก/คลาสย่อย)เป็นคลาสที่รับมรดกมาจากแม่ แต่เพิ่ม "ความพิเศษ" เข้าไปเอง เช่น รถสปอร์ตที่สืบทอดมาจากรถยนต์ จะมีฟังก์ชัน turboBoost() เพิ่มขึ้นมา ซึ่งรถยนต์ทั่วไปไม่มี //

**ประโยชน์การนำโค้ดกลับมาใช้ใหม่ (Code Reusability)** //การสืบทอดช่วยให้นักพัฒนาทำงานได้เร็วขึ้นและผิดพลาดน้อยลง
ลดความซ้ำซ้อน เขียนโค้ดส่วนกลางไว้ที่เดียว (ที่คลาสแม่)
ง่ายต่อการบำรุงรักษา หากต้องการเปลี่ยนตรรกะพื้นฐานของรถยนต์ทุกประเภท เราแก้ที่คลาสแม่จุดเดียว รถลูกๆ ทุกคันจะเปลี่ยนตามโดยอัตโนมัติ
ขยายระบบได้ง่าย หากพรุ่งนี้ต้องการเพิ่ม "รถบรรทุก" เราก็แค่สร้างคลาสใหม่มาสืบทอดจาก "รถยนต์" แล้วเพิ่มฟังก์ชันบรรทุกของเข้าไป //

บทที่ 5 การพหุสัณฐาน (Polymorphism)
**การเขียนทับชื่อ (Method Overloading)** //Overloading คือการสร้างเมธอดที่มี "ชื่อเดียวกัน" ไว้ในคลาสเดียว แต่สิ่งที่ทำให้พวกมันต่างกันคือ พารามิเตอร์ (จำนวนหรือชนิดของข้อมูลที่ไม่เหมือนกัน)
1 เปรียบเทียบง่ายๆ เหมือนปุ่ม "จ่ายเงิน" บนหน้าเว็บ ถ้าคุณส่ง "เลขบัตรเครดิต" เข้าไป ระบบจะตัดบัตร แต่ถ้าคุณส่ง "เบอร์โทรศัพท์" เข้าไป ระบบจะตัดเงินผ่าน Wallet
2 ระบบที่ฉลาด ระบบจะเลือกฟังก์ชันที่เหมาะสมกับข้อมูลที่ส่งมาให้เอง แม้จะเป็นปุ่มที่มีชื่อเหมือนกัน
3 ประโยชน์ ช่วยให้โค้ดอ่านง่าย ไม่ต้องตั้งชื่อเมธอดแยกย่อยจนน่ารำคาญ เช่น แทนที่จะใช้ addInt(), addDouble() เราแค่ใช้ add() ตัวเดียวจบ //

**การเขียนทับนิยาม (Method Overriding)**//Overriding จะเกิดขึ้นเมื่อมีกระบวนการสืบทอด (Inheritance) โดยคลาสลูกทำการ "เขียนคำสั่งทับ" เมธอดที่มีอยู่แล้วในคลาสแม่ เพื่อให้ทำงานได้เฉพาะเจาะจงขึ้น
1 ตัวอย่าง คลาสแม่ชื่อ Animal มีเมธอด makeSound() ที่สั่งให้ส่งเสียงทั่วไป แต่เมื่อคลาส Dog สืบทอดไป มันจะทำการ Override ให้เป็นเสียง "โฮ่งๆ" ส่วนคลาส Cat ก็จะ Override ให้เป็นเสียง "เมี๊ยวๆ"
2 จุดสำคัญ ชื่อเมธอดและพารามิเตอร์ต้อง เหมือนเดิมทุกประการ แต่สิ่งที่เปลี่ยนไปคือ "ตรรกะข้างใน" (Implementation) //

**พลังที่แท้จริง: โค้ดที่ยืดหยุ่นและไร้ if-else** //ประโยชน์ที่ทรงพลังที่สุดคือการทำให้เราสามารถจัดการกับกลุ่มของวัตถุที่หลากหลายได้ด้วยโค้ดชุดเดียว
1 สถานการณ์ ในระบบ CSLearning ของคุณ ถ้าคุณมีรายการของ Question (ข้อสอบ) ซึ่งข้างในอาจจะมีทั้ง MultipleChoiceQuestion (ปรนัย) และ CodingQuestion (อัตนัย)
2 ถ้าไม่มี Polymorphism คุณต้องเขียน if-else เช็คประเภทข้อสอบวุ่นวาย และถ้ามีข้อสอบประเภทใหม่มา คุณต้องกลับมาแก้โค้ดเดิมซ้ำๆ
3 เมื่อมี Polymorphism คุณแค่สั่ง question.checkAnswer() คำเดียวสั้นๆ ในลูป แล้วให้วัตถุแต่ละตัวไปจัดการ "ตรวจคำตอบ" ในแบบของตัวเองตามที่มันถูก Override ไว้ //

บทที่ 6 อินเทอร์เฟซและคลาสบทคัดย่อ (Interfaces and Abstract Classes)

**คลาสบทคัดย่อ (Abstract Classes)** //พิมพ์เขียวที่ยังสร้างไม่เสร็จ:Abstract Class คือคลาสที่ไม่สามารถนำไปสร้างเป็นวัตถุ (Object) ได้โดยตรง แต่มันมีไว้เพื่อให้คลาสอื่นมาสืบทอด (Inherit) ต่อไปเท่านั้น
1 จุดเด่น มันเป็น "ลูกผสม" คือสามารถมีทั้งเมธอดที่มีโค้ดทำงานจริง (Concrete Method) เพื่อแบ่งปันให้ลูกๆ และมีเมธอดเปล่าๆ (Abstract Method) เพื่อบังคับให้ลูกไปเขียนเอง
2 ตัวอย่าง หากคุณทำคลาสแม่ชื่อ BaseAI คุณอาจจะเขียนโค้ดส่วน logHistory() ไว้ให้ AI ทุกตัวใช้เหมือนกัน แต่คุณทิ้งเมธอด generateResponse() ให้เป็น Abstract เพื่อให้ AI แต่ละตัวไปเขียนตรรกะของตัวเอง //

**อินเทอร์เฟซ (Interfaces)** //สัญญาใจที่บิดพริ้วไม่ได้:Interface ก้าวไปไกลกว่า Abstract Class เพราะมันคือ "สัญญา" (Contract) บริสุทธิ์ มันจะไม่บอกเลยว่าต้องทำงานยังไง แต่มันจะบอกว่า "ถ้าคุณอยากจะเป็นส่วนหนึ่งของระบบนี้ คุณต้องทำสิ่งเหล่านี้ให้ได้นะ"
1 กฎเหล็ก ในภาษาส่วนใหญ่ Interface จะไม่มีการเก็บสถานะ (Variables) และไม่มีโค้ดการทำงาน ทุกอย่างคือเมธอดว่างเปล่าที่รอการเติมเต็ม
2 ตัวอย่าง อินเทอร์เฟซ CloudStorage อาจจะระบุว่าต้องมีฟังก์ชัน upload() และ download() ไม่ว่าคุณจะใช้ Google Drive หรือ Dropbox ตราบใดที่ทำตามสัญญานี้ โค้ดส่วนกลางของคุณก็พร้อมจะเรียกใช้งานได้ทันที //

**การแก้ปัญหา Multiple Inheritance และ Loose Coupling** //ทำไมเราถึงต้องมีทั้งสองอย่าง? คำตอบอยู่ในเชิงวิศวกรรมครับ
1 Multiple Inheritance ภาษาอย่าง Java หรือ C# ไม่อนุญาตให้สืบทอดจากคลาสแม่หลายตัวเพื่อป้องกันความสับสน แต่เราสามารถ Implement หลาย Interface พร้อมกันได้ (เช่น เป็นทั้ง Drawable และ Clickable)
2 Loose Coupling (การลดการผูกมัด) การออกแบบโดยพึ่งพา Interface ช่วยให้คลาสต่างๆ ไม่ต้องรู้จักตัวตนที่แท้จริงของกันและกัน ขอแค่ทำตามสัญญาได้ก็พอ ทำให้เวลาเปลี่ยนระบบหลังบ้าน โค้ดส่วนหน้าจะไม่พังตามไปด้วย //


บทที่ 7 ความสัมพันธ์ของวัตถุเชิงลึก (Object Relationships: Association, Aggregation, and Composition)
**Association (ความสัมพันธ์แบบรู้จักกัน)** //นี่คือความสัมพันธ์ระดับที่หลวมที่สุด วัตถุสองสิ่งมีความเกี่ยวข้องกันเพียงเพื่อ "ใช้บริการ" หรือ "สื่อสาร" กันเท่านั้น โดยที่ทั้งคู่มีอิสระต่อกันโดยสมบูรณ์
1 ลักษณะเด่น วัตถุชิ้นหนึ่งอาจจะมีตัวแปรที่อ้างอิง (Reference) ไปยังอีกวัตถุหนึ่ง แต่ต่างฝ่ายต่างเกิดและตายแยกกันได้
2 ตัวอย่าง แพทย์ (Doctor) กับ คนไข้ (Patient) แพทย์สามารถรักษาคนไข้หลายคน และคนไข้ก็สามารถไปหาแพทย์หลายท่าน หากแพทย์ลาออก คนไข้ก็ยังคงมีตัวตนอยู่ในระบบ และในทางกลับกันด้วยครับ //

**Aggregation (ความสัมพันธ์แบบรวมกลุ่ม - "Weak HAS-A")** //เข้มข้นขึ้นมาอีกนิด คือความสัมพันธ์แบบ "ส่วนหนึ่งของ" (Part-of) แต่เป็นความสัมพันธ์ที่ไม่ได้ยึดติดกันแบบตายตัว
1 ลักษณะเด่น วัตถุที่เป็นส่วนประกอบ (Part) สามารถอยู่รอดได้แม้ว่าวัตถุที่เป็นเจ้าของ (Whole) จะถูกลบออกไปจากหน่วยความจำแล้วก็ตาม
2 ตัวอย่าง มหาวิทยาลัย (University) กับ นักศึกษา (Student) นักศึกษาคือส่วนหนึ่งของมหาวิทยาลัย แต่ถ้ามหาวิทยาลัยปิดตัวลง นักศึกษาก็ไม่ได้ "หาย" ไปจากโลกนี้ เขายังคงเป็นบุคคลที่มีตัวตนและสามารถย้ายไปสังกัดมหาวิทยาลัยอื่นได้ //

**Composition (ความสัมพันธ์แบบส่วนประกอบ - "Strong HAS-A")** //นี่คือระดับความสัมพันธ์ที่เข้มข้นที่สุดและเด็ดขาดที่สุด เป็นการเป็นเจ้าของแบบที่ "แยกจากกันไม่ได้"
1 ลักษณะเด่น วัตถุที่เป็นส่วนประกอบย่อยจะถูกสร้างและถูกทำลายไปพร้อมกับวัตถุที่เป็นเจ้าของเสมอ (Co-dependent Lifecycle) หากวัตถุหลัก (Container) ตาย วัตถุลูก (Components) ต้องตายตาม
2 ตัวอย่าง ไฟล์เอกสาร (Document) กับ หน้ากระดาษ (Page) หากคุณลบไฟล์เอกสารทิ้ง หน้ากระดาษทุกหน้าที่อยู่ในไฟล์นั้นจะต้องสูญสลายไปพร้อมกัน เพราะหน้ากระดาษเหล่านั้นไม่มีความหมายหรือตัวตนอยู่ภายนอกไฟล์เอกสารนั้นได้ //

บทที่ 8 การจัดการข้อผิดพลาดระดับวัตถุ (Object-Oriented Exception Handling)

**โครงสร้าง Try-Catch-Finally ตาข่ายนิรภัยของโปรแกรม** //ในการรับมือกับความเสี่ยง เราใช้โครงสร้างที่เปรียบเสมือนการขับรถที่มีถุงลมนิรภัย
1 try (ลอง) ส่วนที่รวบรวมโค้ดที่มีความเสี่ยงจะเกิดข้อผิดพลาด เช่น การต่อฐานข้อมูลร้านปลาสวยงามของคุณ หรือการคำนวณราคา
2 catch (ดัก) เปรียบเสมือนแผนสำรอง หากเกิดข้อผิดพลาดใน try โปรแกรมจะกระโดดมาที่นี่เพื่อแก้ไขสถานการณ์โดยไม่หยุดทำงาน (Crash)
3 finally (สุดท้าย) ส่วนที่จะทำงาน เสมอ ไม่ว่าจะเกิดข้อผิดพลาดหรือไม่ มักใช้สำหรับการ "ปัดกวาด" ทรัพยากร เช่น การปิดการเชื่อมต่อฐานข้อมูลเพื่อป้องกัน Memory Leak //

**ลำดับชั้นของข้อผิดพลาด (Exception Hierarchy)** //ข้อผิดพลาดใน OOP ถูกจัดเป็นลำดับชั้นเหมือนคลาสปกติ เพื่อให้เราเลือกจัดการได้อย่างเหมาะสม
1 Checked Exceptions ข้อผิดพลาดที่ระบบบังคับให้เราเตรียมรับมือ (เช่น หาไฟล์รูปภาพปลาไม่เจอ)
2 Unchecked / Runtime Exceptions ข้อผิดพลาดที่มักเกิดจากลอจิกการเขียนโปรแกรม (เช่น การเข้าถึงลำดับในลิสต์ที่ไม่มีอยู่จริง)
3 กลยุทธ์การดักจับ นักพัฒนาที่ดีควรดักจับข้อผิดพลาดที่ เฉพาะเจาะจงที่สุด (Specific) ก่อนข้อผิดพลาดแบบเหมาเข่ง เพื่อให้รู้ต้นตอที่แท้จริง //

**การโยนและการออกแบบข้อผิดพลาดเอง (Throw & Custom Exceptions)** //ในบางครั้ง เราไม่ได้ต้องการดักจับข้อผิดพลาดด้วยตัวเอง แต่ต้องการแจ้งให้ส่วนอื่นทราบ
1 throw เราสามารถ "โยน" วัตถุข้อผิดพลาดออกไปเองเมื่อพบความผิดปกติ เช่น หากยอดสั่งซื้อปลาเป็นค่าลบ เราสามารถสั่ง throw new InvalidOrderException() ได้ทันที
2 Custom Exceptions การสร้างคลาสข้อผิดพลาดเองช่วยให้โค้ดสื่อความหมายได้ชัดเจน เช่น แทนที่จะบอกแค่ว่า "ผิดพลาด" เราบอกได้เลยว่า "สินค้าในคลังไม่พอ" ผ่าน OutOfStockException //

บทที่ 9 พื้นฐาน Design Patterns (Fundamentals of Design Patterns)
**Singleton Pattern มีเพียงหนึ่งเดียวในจักรวาล** //บางครั้งเราต้องการให้วัตถุชิ้นหนึ่งมีเพียง "ชิ้นเดียว" ตลอดการรันโปรแกรม เพื่อป้องกันการแย่งทรัพยากรหรือความสับสนของข้อมูล1 แนวคิด ควบคุมให้คลาสสามารถสร้าง Instance ได้เพียงตัวเดียวเท่านั้น และเปิดช่องทางให้ส่วนอื่นของโปรแกรมเข้าถึงได้จากจุดเดียว (Global Access)2 เคสการใช้งานจริง เช่น Database Connection หรือ Configuration Manager ที่ทุกส่วนของแอปฯ ต้องมาดึงค่าจากที่เดียวกัน3 ข้อควรระวัง หากใช้มากเกินไปอาจทำให้โค้ดผูกมัดกันแน่น (Tight Coupling) และทดสอบได้ยากครับ //

**Factory Method Pattern โรงงานผลิตวัตถุ** //เมื่อตรรกะการสร้างวัตถุซับซ้อน หรือเรายังไม่รู้แน่ชัดว่าในอนาคตจะมีวัตถุประเภทไหนเพิ่มเข้ามาบ้าง เราจะไม่สั่ง new Object() ตรงๆ แต่จะใช้โรงงานเข้ามาช่วย
1 แนวคิด สร้าง "เมธอดโรงงาน" ขึ้นมาเพื่อทำหน้าที่สร้างวัตถุแทนเรา โดยผู้เรียกไม่ต้องสนใจวิธีสร้างหรือคลาสย่อยที่ใช้ ขอแค่ได้วัตถุที่ทำงานตาม Interface ที่กำหนดไว้
2 เคสการใช้งานจริง ระบบการส่งข้อความ (Email, SMS) ที่ในอนาคตอาจมี Line หรือ WhatsApp เพิ่มเข้ามา เราแค่เพิ่มประเภทในโรงงานโดยไม่ต้องแก้โค้ดหลัก //

**ตารางเปรียบเทียบ Singleton vs Factory Method** //คุณสมบัติSingletonFactory Methodเป้าหมายหลักจำกัดจำนวนวัตถุให้เหลือ 1 เดียวแยกตรรกะการสร้างวัตถุออกจากโค้ดที่เรียกใช้ความยืดหยุ่นต่ำ (เน้นความนิ่งของสถานะ)สูง (รองรับการเพิ่มคลาสใหม่ๆ)จุดเน้นการเข้าถึง (Access Control)การสร้าง (Creation Logic)//

**ทำไมต้องใช้ Design Patterns?** //การทำความเข้าใจแพทเทิร์นเหล่านี้จะช่วยยกระดับคุณไปสู่การเป็น "สถาปนิกซอฟต์แวร์" (Software Architect)1 ภาษากลาง เมื่อบอกทีมว่า "ตรงนี้ใช้ Singleton นะ" ทุกคนจะเห็นภาพโครงสร้างเดียวกันทันทีโดยไม่ต้องอธิบายยาว2 ความทนทาน โค้ดที่ออกแบบตามแพทเทิร์นจะรองรับการขยายตัว (Scalability) ได้ดีกว่า และมีบั๊กน้อยกว่าการออกแบบเองแบบไม่มีหลักการ //

บทที่ 10 หลักการ SOLID (The Golden Rules of Software Engineering)

**S Single Responsibility Principle (SRP)** //"หนึ่งคน หนึ่งหน้าที่" อย่าสร้างคลาสที่เป็น "เป็ด" ที่ทำได้ทุกอย่าง
1 ตัวอย่าง คลาส Report ไม่ควรมีหน้าที่ทั้งคำนวณข้อมูล, จัดฟอร์แมต JSON และส่งอีเมล
2 ทำไมต้องแยก? หากคุณต้องการเปลี่ยนรูปแบบอีเมล คุณไม่ควรต้องเสี่ยงทำให้ลอจิกการคำนวณพัง การแยกหน้าที่ทำให้โค้ดอ่านง่ายและหาบั๊กได้ง่ายขึ้นครับ //

**O Open/Closed Principle (OCP)**//"ขยายได้แต่ห้ามแก้" หัวใจคือการใช้ Interface หรือ Abstract Class เข้ามาช่วย
1 การทำงาน เมื่อต้องการเพิ่มวิธีการชำระเงินแบบใหม่ เราควรสร้างคลาสใหม่มาสืบทอดมาตรฐานเดิม
2 ข้อดี ไม่ต้องเข้าไปเขียน if-else เพิ่มในคลาสหลักที่ทำงานอยู่แล้ว ซึ่งเสี่ยงต่อการทำฟีเจอร์เก่าพัง (Regression Bug) //

**L Liskov Substitution Principle (LSP)** //"ลูกต้องเก่งเท่าแม่ (ในทางที่แม่กำหนด)" คลาสลูกต้องแทนที่แม่ได้สมบูรณ์
1 ตัวอย่างที่ผิด ถ้าคลาสแม่บอกว่า Bird สามารถ fly() ได้ คลาสลูกอย่าง Penguin ที่สืบทอดไปแต่บินไม่ได้จะทำให้โปรแกรมพัง
2 กฎเหล็ก คลาสลูกต้องถูกนำไปวางแทนที่คลาสแม่ได้ โดยที่ลอจิกโดยรวมยังคงถูกต้องและไม่เกิดพฤติกรรมประหลาด //

**I Interface Segregation Principle (ISP)** //"อินเทอร์เฟซจิ๋วแต่แจ๋ว" อย่าสร้างสัญญาที่ใหญ่เกินความจำเป็น
1 ปัญหา Interface ขนาดใหญ่จะบังคับให้คลาสที่นำไปใช้ต้องเขียนโค้ดว่างเปล่า (Empty Methods) เพื่อรองรับฟังก์ชันที่ไม่ได้ใช้
2 ทางออก เราควรแยกเป็น Interface ย่อยๆ ที่เฉพาะเจาะจง เพื่อให้คลาสต่างๆ เลือกใช้เฉพาะ "สัญญา" ที่เกี่ยวข้องจริงๆ เท่านั้น //

**D Dependency Inversion Principle (DIP)** //"พึ่งพาสิ่งที่จับต้องไม่ได้" เพื่อความยืดหยุ่นสูงสุด
1 แนวคิด คลาสระดับสูงไม่ควรเรียกใช้งานคลาสระดับต่ำโดยตรง แต่ทั้งคู่ควรคุยกันผ่าน Interface
2 เปรียบเทียบ เหมือนปลั๊กไฟ (Abstraction) เราไม่สนว่าไฟฟ้ามาจากไหน (Concrete) ขอแค่ปลั๊กเสียบกันได้ก็พอ
3 ผลลัพธ์ ช่วยให้เราสลับเปลี่ยนเทคโนโลยีเบื้องหลัง (เช่น เปลี่ยน Database) ได้โดยไม่ต้องรื้อโค้ดหลักใหม่ทั้งหมด //

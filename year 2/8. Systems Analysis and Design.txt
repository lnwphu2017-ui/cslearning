8. Systems Analysis and Design
บทที่ 1: วงจรชีวิตซอฟต์แวร์ (System Development Life Cycle - SDLC)
**ขั้นตอนมาตรฐานของ SDLC (6 ขั้นตอนหลัก)**
//เพื่อให้การพัฒนาระบบเป็นไปอย่างราบรื่น วิศวกรซอฟต์แวร์มักจะเดินตามเข็มทิศ 6 ขั้นตอนนี้ประกอบด้วย:
Planning / Investigation การตั้งคำถามถึงปัญหาและความคุ้มค่าในการลงทุน (Feasibility Study)
Analysis การรวบรวมความต้องการ (Requirement Elicitation) เพื่อระบุสิ่งที่ระบบต้องทำได้ (Functional Requirements)
Design การวางพิมพ์เขียว ทั้งโครงสร้างฐานข้อมูล (ER-Diagram), หน้าตาโปรแกรม (UI/UX) และสถาปัตยกรรมระบบ
Development การลงมือเขียนโค้ดตามการออกแบบ
Testing การตรวจสอบหาข้อผิดพลาด (Bugs) และทดสอบความถูกต้องตามความต้องการที่วิเคราะห์ไว้
Deployment & Maintenance การติดตั้งระบบให้ผู้ใช้จริง และคอยดูแลรักษาปรับปรุงตาม Feedback//

**โมเดลการพัฒนา: Waterfall vs. Agile**
//แนวคิดในการขับเคลื่อน SDLC ที่นิยมนำมาเปรียบเทียบกันมากที่สุดคือ:
Waterfall Model (แบบน้ำตก) เน้นความชัดเจนและต้องทำทีละขั้นตอนให้เสร็จสมบูรณ์ก่อนย้ายไปขั้นถัดไป เหมาะสำหรับโปรเจกต์ที่ความต้องการชัดเจนและเปลี่ยนแปลงยาก แต่ข้อเสียคือหากมีความเข้าใจคลาดเคลื่อนตั้งแต่ต้น การแก้ไขตอนท้ายจะยากและมีค่าใช้จ่ายสูง
Agile Methodology (แบบคล่องตัว) เน้นการแบ่งงานเป็นส่วนย่อย (Sprints) และทำงานวนซ้ำ (Iterative) เพื่อส่งมอบซอฟต์แวร์ที่ใช้งานได้จริงให้ผู้ใช้ดูบ่อยๆ เหมาะสำหรับซอฟต์แวร์ยุคใหม่ที่ต้องการความรวดเร็วและพร้อมรับความเปลี่ยนแปลง ช่วยลดความเสี่ยงในการสร้างสิ่งที่ไม่ตอบโจทย์ผู้ใช้จริง//

บทที่ 2: การสืบค้นความต้องการ (Requirements Elicitation - ศิลปะการจับใจความผู้ใช้งาน)
**เทคนิคการรวบรวมข้อมูล (Requirements Elicitation Techniques)**
//เรามีหลายวิธีในการดึงข้อมูลจากผู้ใช้งาน ซึ่งแต่ละวิธีมีข้อดีและข้อเสียที่แตกต่างกันไปตามบริบทของงานประกอบด้วย:
Interviews (การสัมภาษณ์) วิธีที่ช่วยเจาะลึกถึงปัญหาและอารมณ์ความรู้สึกได้ดีที่สุด เหมาะสำหรับการเก็บข้อมูลเชิงลึกจากผู้บริหารหรือบุคคลสำคัญในองค์กร
Questionnaires (แบบสอบถาม) เหมาะสำหรับรวบรวมข้อมูลจากกลุ่มคนจำนวนมากในเวลาอันสั้น เช่น การสำรวจความต้องการของนักศึกษาเพื่อพัฒนา CSLearning เพื่อให้ได้ข้อมูลเชิงสถิติที่ชัดเจน
Joint Application Design (JAD) การนำ User, Designer และ Developer มาประชุมเชิงปฏิบัติการร่วมกัน เพื่อหาข้อสรุปที่ทุกคนเห็นพ้องในทันที ช่วยลดปัญหาการสื่อสารที่คลาดเคลื่อนได้มหาศาล//

**การแยกแยะประเภทความต้องการ (Functional vs. Non-functional)**
//การจัดหมวดหมู่ความต้องการเป็นเรื่องสำคัญที่นักวิเคราะห์ระบบต้องคัดแยกให้ขาดเพื่อประสิทธิภาพในการออกแบบ:
Functional Requirements (ฟังก์ชันหลัก) ระบุว่า "ระบบต้องทำอะไรได้บ้าง" (What) เช่น ระบบต้องสามารถดึงไฟล์ PDF จาก Google Drive มาทำ RAG ได้ หรือต้องแสดงผลคะแนนทันทีหลังส่งข้อสอบ
Non-functional Requirements (คุณลักษณะเสริม) ระบุว่า "ระบบต้องมีคุณภาพอย่างไร" (How) เช่น ระบบต้องรองรับผู้ใช้พร้อมกัน 1,000 คน โดยไม่หน่วง (Scalability) หรือต้องโหลดหน้าแรกเสร็จภายใน 2 วินาที (Performance)//

**ปัญหา "ความต้องการที่ไม่ชัดเจน" (Hidden Requirements)**
//ความท้าทายหลักคือการเปลี่ยนความต้องการที่เลื่อนลอยให้กลายเป็นสเปกที่จับต้องได้
Requirement Refinement หน้าที่ของนักพัฒนาคือการเปลี่ยนคำว่า "ใช้งานง่าย" ของผู้ใช้ ให้กลายเป็นข้อกำหนดทางเทคนิค เช่น "ผู้ใช้ต้องสามารถเข้าถึงฟีเจอร์หลักได้ภายใน 3 คลิก"
Precision การระบุสเปกที่ชัดเจนจะช่วยลดความขัดแย้งในขั้นตอนการตรวจรับงาน และช่วยให้ทีมพัฒนาสามารถสร้างระบบได้ตรงตามความคาดหวังจริง//

บทที่ 3: Data Flow Diagrams (แผนภาพกระแสข้อมูล - พิมพ์เขียวการเดินทางของข้อมูล)
**สัญลักษณ์มาตรฐานใน DFD (Gane & Sarson หรือ Yourdon & Coad)**
//สัญลักษณ์สากล 4 อย่างที่เป็นหัวใจหลักในการสื่อสารประกอบด้วย
External Entity (หน่วยงานภายนอก) สี่เหลี่ยมแทนสิ่งที่อยู่นอกระบบแต่มีการแลกเปลี่ยนข้อมูล เช่น นักศึกษา หรือ API ภายนอก
Process (กระบวนการ) สี่เหลี่ยมมุมมนหรือวงกลมแทนกิจกรรมที่เปลี่ยนรูปข้อมูล เช่น "คำนวณคะแนน" หรือ "ค้นหาข้อมูลด้วย RAG"
Data Store (แหล่งเก็บข้อมูล) เส้นขนานหรือสี่เหลี่ยมเปิดด้านหนึ่งแทนที่พักข้อมูลเพื่อรอการใช้งาน เช่น Database ใน Supabase
Data Flow (กระแสน้ำข้อมูล) เส้นลูกศรที่บอกทิศทางการเคลื่อนที่ของข้อมูลจากจุดหนึ่งไปอีกจุดหนึ่ง//

**ระดับของ DFD (Leveling)**
//การเขียน DFD มักใช้แนวคิด Top-Down Approach เพื่อความชัดเจนในการวิเคราะห์
Context Diagram (ระดับ 0) มองระบบทั้งหมดเป็นเพียง "กระบวนการเดียว" เพื่อระบุขอบเขตระบบ (System Boundary) และสิ่งภายนอกที่เกี่ยวข้อง เช่น ระบบ CSLearning ที่รับคำถามจากนักศึกษาและส่งคำตอบกลับไป
DFD Level 0 (หรือ Level 1) การ "ระเบิด" (Explode) กระบวนการหลักออกมาเป็นขั้นตอนย่อยๆ ภายในระบบ เช่น การแยกขั้นตอน "ตรวจสอบสิทธิ์" ออกจาก "การประมวลผลด้วย AI" เพื่อให้เห็นการทำงานเชิงลึกมากขึ้น//

**กฎเหล็กในการเขียน DFD**
//เพื่อความถูกต้องตามหลักการวิศวกรรมข้อมูล นักพัฒนาต้องยึดกฎสำคัญดังนี้
ห้ามไหลตรงระหว่าง Entity ข้อมูลจาก External Entity จะส่งหาคู่สนทนาภายนอกโดยตรงไม่ได้ ต้องผ่าน Process เสมอ
ห้ามไหลตรงระหว่าง Data Store ข้อมูลจะย้ายจากฐานข้อมูลหนึ่งไปอีกฐานข้อมูลหนึ่งโดยไม่ผ่านการประมวลผล (Process) ไม่ได้
หลีกเลี่ยง Black Hole ทุกกระบวนการต้องมีทั้ง "ข้อมูลเข้า" และ "ข้อมูลออก" หากข้อมูลไหลเข้าแล้วหายไปโดยไม่มีผลลัพธ์ออกมาจะถือว่าผิดหลักการทำงานของระบบซอฟต์แวร์//

บทที่ 4: พจนานุกรมข้อมูลและความสมบูรณ์ (Data Dictionary - คลังคำอธิบายข้อมูล)
**Metadata ข้อมูลที่อธิบายข้อมูล**
//พจนานุกรมข้อมูลทำหน้าที่จัดเก็บ Metadata เพื่อระบุคุณสมบัติของข้อมูลทุกตัวที่ไหลในระบบ (Data Flow) และที่ถูกจัดเก็บ (Data Store) ประกอบด้วย
Data Element Name ชื่อที่ใช้เรียกข้อมูล เช่น student_id หรือ question_content
Data Type ชนิดของข้อมูล เช่น String, Integer หรือ Boolean
Length / Format ความยาวหรือรูปแบบที่กำหนด เช่น VARCHAR(50) หรือ YYYY-MM-DD
Validation Rules กฎตรวจสอบความถูกต้อง เช่น คะแนนต้องอยู่ในช่วง 0-100 เท่านั้น
Description คำอธิบายหน้าที่และความหมายของข้อมูลนั้นๆ เพื่อป้องกันความเข้าใจผิด//

**ความสมบูรณ์ของข้อมูล (Data Integrity)**
//การกำหนดพจนานุกรมข้อมูลที่ชัดเจนส่งผลดีต่อความมั่นคงของระบบในระยะยาว:
Data Consistency มั่นใจได้ว่าข้อมูลจะมีสเปกเดียวกันเสมอไม่ว่าจะไหลผ่าน Process ใด ช่วยลดโอกาสเกิด Error จากการเปลี่ยน Data Type ผิดพลาด
Referential Integrity ช่วยให้เห็นความสัมพันธ์ระหว่างข้อมูล (Foreign Keys) ได้ชัดเจน ลดโอกาสที่จะเกิดข้อมูลที่ไม่มีความสัมพันธ์หลงเหลืออยู่ในระบบ (Orphan Data)//

**การนิยามโครงสร้างข้อมูล (Data Structure Notation)**
//เราใช้สัญลักษณ์ทางคณิตศาสตร์พื้นฐานเพื่ออธิบายโครงสร้างข้อมูลที่ซับซ้อนให้กลายเป็นข้อกำหนดที่จับต้องได้:
= หมายถึง ประกอบด้วย (Composed of)
+ หมายถึง และ (And)
[ | ] หมายถึง เลือกอย่างใดอย่างหนึ่ง (Either/Or)
{ } หมายถึง ข้อมูลที่เกิดขึ้นซ้ำ (Iterations)
( ) หมายถึง ข้อมูลที่มีหรือไม่มีก็ได้ (Optional)
ตัวอย่างโครงสร้างข้อมูล
Student_Profile = First_Name + Last_Name + (Middle_Name) + {Phone_Number}
(หมายถึง: โปรไฟล์นักศึกษาประกอบด้วยชื่อ, นามสกุล, ชื่อกลางจะมีหรือไม่ก็ได้ และสามารถมีเบอร์โทรศัพท์ได้หลายเบอร์)//


บทที่ 5: OO Analysis (การวิเคราะห์เชิงวัตถุ - การมองระบบผ่านกลุ่มของวัตถุ)
**UML (Unified Modeling Language) ภาษาสากลของการสื่อสาร**
//เพื่อให้ทีมพัฒนามองเห็นภาพตรงกัน เราจึงใช้แผนภาพ UML เป็นมาตรฐานในการจำลองระบบ ซึ่งในระยะการวิเคราะห์จะมีแผนภาพหลัก 2 ตัวที่ต้องให้ความสำคัญ://

**A. Use Case Diagram (มุมมองพฤติกรรม)**
//ทำหน้าที่อธิบายปฏิสัมพันธ์ระหว่างผู้ใช้กับระบบเพื่อระบุขอบเขต (Scope) ของงาน
Actor ผู้ใช้งานหรือระบบภายนอก (แทนด้วยรูปคน)
Use Case กิจกรรมหรือฟังก์ชันที่ระบบทำงานให้ (แทนด้วยรูปวงรี)
Boundary เส้นขอบเขตที่คัดแยกระหว่างสิ่งที่ระบบทำ กับสิ่งที่อยู่นอกระบบ//

**B. Class Diagram (มุมมองโครงสร้าง)**
//ทำหน้าที่ระบุโครงสร้างและรายละเอียดของวัตถุที่จะถูกสร้างขึ้นจริงในระบบ ประกอบด้วย:
Class Name ชื่อคลาส เช่น Student หรือ AIModel
Attributes ข้อมูลหรือคุณสมบัติภายใน เช่น studentID, name
Operations (Methods) พฤติกรรมที่คลาสนั้นทำได้ เช่น submitQuestion()
Relationships เส้นความสัมพันธ์ที่บอกว่าคลาสต่างๆ เชื่อมโยงกันอย่างไร เช่น การสืบทอดคุณสมบัติ (Inheritance)//

**หัวใจของ OO เพื่อการนำกลับมาใช้ใหม่ (Reusability)**
//การวิเคราะห์เชิงวัตถุช่วยให้นักพัฒนาได้รับประโยชน์มหาศาลผ่าน 3 แนวคิดหลัก:
Encapsulation การห่อหุ้มและซ่อนรายละเอียดภายในไว้ภายในวัตถุ ช่วยลดผลกระทบต่อส่วนอื่นเมื่อมีการแก้ไขโค้ดภายในคลาสนั้นๆ
Inheritance การสืบทอดคุณสมบัติจากคลาสแม่สู่คลาสลูก ช่วยให้เราไม่ต้องเขียนโค้ดซ้ำซ้อนและสร้างส่วนขยายได้รวดเร็ว
Polymorphism ความสามารถในการประมวลผลคำสั่งเดียวกันด้วยวิธีการที่ต่างกันไปตามประเภทของวัตถุ เช่น คำสั่ง calculateScore() อาจทำงานต่างกันระหว่าง "ปรนัย" และ "อัตนัย"//

บทที่ 6: UML Behavior Models (การจำลองพฤติกรรม - เมื่อระบบเริ่มเคลื่อนไหว)
**Sequence Diagram ลำดับการโต้ตอบตามกาลเวลา**
//เป็นเครื่องมือสำคัญสำหรับนักพัฒนา เพราะช่วยให้เห็นว่าวัตถุแต่ละตัว "คุย" กันอย่างไรตามลำดับเวลาจากบนลงล่าง:
Lifelines เส้นประแนวดิ่งแทนตัวตนของวัตถุในช่วงเวลานั้นๆ
Messages ลูกศรส่งไปมาระหว่างเส้นประ เพื่อเรียกใช้งาน Method หรือส่งค่ากลับ (Return Value)
Activation Bar แท่งสี่เหลี่ยมบนเส้นประที่ระบุว่าวัตถุตัวนั้นกำลัง "ทำงาน" หรือประมวลผลอยู่
ประโยชน์ ช่วยให้เห็น Flow การเรียก API เช่น เมื่อ User ส่งคำถาม AI Agent จะไปคุยกับ Vector DB อย่างไรและส่งผลลัพธ์กลับมาตอนไหน//

**Activity Diagram: แผนภาพกิจกรรมและเวิร์กโฟลว์**
//เปรียบเสมือน Flowchart เวอร์ชันอัปเกรดที่ใช้จำลองลำดับขั้นตอนการทำงาน (Workflow) ของระบบ:
Action States ขั้นตอนกิจกรรมย่อยๆ ในกระบวนการ
Decision Node จุดตัดสินใจเงื่อนไข (สี่เหลี่ยมข้าวหลามตัด) เพื่อแยกทิศทางของ Logic
Fork & Join การแตกงานออกไปทำพร้อมกัน (Parallel) และการรอให้งานทั้งหมดเสร็จก่อนรวมกลับมา
Swimlanes การแบ่งเขตเพื่อระบุว่า "ใคร" (เช่น User, System, หรือ Admin) เป็นผู้รับผิดชอบกิจกรรมนั้นๆ
ประโยชน์ ช่วยให้เห็นภาพรวมของ Business Logic ที่ซับซ้อน เช่น ขั้นตอนการสมัครสมาชิกและการยืนยันตัวตน//

**State Machine Diagram: การจัดการสถานะของวัตถุ**
//ใช้จำลอง "สถานะ" ของวัตถุที่เปลี่ยนไปตามเหตุการณ์ (Event) ที่มากระตุ้น เหมาะสำหรับวัตถุที่มีวงจรชีวิตชัดเจน:
State สถานะของวัตถุในจุดเวลาหนึ่ง เช่น Draft, In Progress, หรือ Completed
Transition เส้นเชื่อมที่บอกว่าวัตถุเปลี่ยนจากสถานะหนึ่งไปอีกสถานะหนึ่งได้อย่างไร
Event/Condition เงื่อนไขหรือเหตุการณ์ที่ต้องเกิดขึ้นก่อนจะเปลี่ยนสถานะได้
ประโยชน์ ป้องกัน Logical Error ในระบบ เช่น ป้องกันการให้คะแนนข้อสอบในขณะที่สถานะยังเป็น Draft หรือยังไม่ได้เริ่มทำ//

บทที่ 7: สถาปัตยกรรมและ UI Design (จากตรรกะสู่การสร้างสรรค์ทางกายภาพ)
**System Architecture: รากฐานที่แข็งแกร่ง**
//การออกแบบสถาปัตยกรรมคือการเลือก "เครื่องยนต์" และ "โครงสร้าง" ให้เหมาะกับภาระงาน (Workload) เพื่อประสิทธิภาพสูงสุด:
Layered Architecture: การแยกส่วนการทำงานเป็นชั้นๆ เช่น Presentation Layer (หน้าเว็บ), Logic Layer (API/AI Agent) และ Data Layer (Database) เพื่อให้ง่ายต่อการแก้ไขเฉพาะส่วนโดยไม่กระทบภาพรวม
Scalability & Cloud-Native การเลือกใช้ Platform อย่าง Vercel หรือ Supabase เพื่อให้ระบบรองรับจำนวนผู้ใช้ที่เพิ่มขึ้นได้โดยอัตโนมัติ และลดภาระในการดูแล Server เอง
Network & Latency การวางแผนเส้นทางรับส่งข้อมูล เช่น การใช้ Edge Computing เพื่อให้ AI ตอบสนองผู้ใช้ได้ลื่นไหลที่สุด//

**User Interface (UI) & User Experience (UX) Design**
//การออกแบบไม่ใช่แค่เรื่องความสวยงาม แต่คือ "การสื่อสาร" เพื่อให้มนุษย์ใช้งานคอมพิวเตอร์ได้อย่างแม่นยำ
Input Design (ส่วนนำเข้า) มุ่งเน้นการลดความพยายามและป้องกันความผิดพลาด (Error Prevention) เช่น การใช้ UI Component ที่เหมาะสม (Dropdown/Radio) และการตรวจสอบความถูกต้อง (Validation) ตั้งแต่ต้นทาง
Output Design (ส่วนแสดงผล) ต้องสรุปข้อมูลที่ซับซ้อนให้เข้าใจง่าย เช่น การใช้ Dashboard หรือกราฟแสดงพัฒนาการการเรียนรู้ แทนการแสดงเพียงตัวเลขดิบๆ//

**หลักการออกแบบที่เน้นผู้ใช้เป็นศูนย์กลาง (User-Centered Design)**
//เพื่อให้เป็นระบบที่เอยากใช้งานจริง ควรยึดหลักสำคัญ 3 ประการ:
Consistency ความสม่ำเสมอขององค์ประกอบ เช่น สี ฟอนต์ และขนาดปุ่ม เพื่อให้ผู้ใช้เกิดความคุ้นเคยและไม่ต้องเรียนรู้ใหม่ในแต่ละหน้า
Feedback ระบบต้องมีการโต้ตอบทุกครั้งที่มีการกระทำ เช่น การแสดง Progress Bar ขณะ AI กำลังประมวลผลคำตอบ เพื่อให้ผู้ใช้รู้ว่าระบบไม่ได้ค้าง
Accessibility (a11y) การออกแบบให้ทุกคนเข้าถึงได้ เช่น การเลือกคู่สีที่มีความต่าง (Contrast) สูง เพื่อให้การอ่านบทเรียนวิชาคอมพิวเตอร์เป็นเรื่องง่ายสำหรับทุกคน//

บทที่ 8: การติดตั้งและขยายระบบ (Implementation & Maintenance - การส่งมอบและความยั่งยืน)
**กลยุทธ์การเปลี่ยนถ่ายระบบ (Conversion Strategies)**
//เมื่อระบบพัฒนาเสร็จสิ้น การนำระบบใหม่เข้าแทนที่วิธีการทำงานเดิมสามารถทำได้หลายรูปแบบตามความเหมาะสมของความเสี่ยงและทรัพยากร:
Direct Cutover (เปลี่ยนทันที) เลิกใช้วิธีเดิมและเปิดใช้ระบบใหม่ทันที มีข้อดีคือรวดเร็วและประหยัด แต่มีความเสี่ยงสูงที่สุดหากเกิดข้อผิดพลาดรุนแรง
Parallel Conversion (ใช้งานคู่ขนาน) รันระบบเก่าและใหม่ไปพร้อมกัน ปลอดภัยที่สุดเพราะมีระบบสำรองเสมอ แต่ต้องใช้ทรัพยากรสูงและผู้ใช้ต้องทำงานซ้ำซ้อน
Pilot Conversion (ระบบนำร่อง) ทดลองใช้กับกลุ่มผู้ใช้ขนาดเล็กก่อน (เช่น เฉพาะกลุ่มเพื่อนในเซกชัน) เพื่อจำกัดวงกระทบหากเกิดปัญหา ก่อนจะขยายผลสู่กลุ่มใหญ่
Phased Conversion (ทยอยเปิดใช้งาน) ค่อยๆ เปิดฟีเจอร์ทีละส่วนตามลำดับความสำคัญ ช่วยให้ผู้ใช้ไม่ต้องเรียนรู้ระบบใหม่ทั้งหมดในครั้งเดียว//

**การดูแลรักษาซอฟต์แวร์ (Software Maintenance)**
//งานของนักพัฒนาไม่ได้จบลงที่การส่งมอบ แต่คือการดูแลให้ระบบทำงานได้ดีอยู่เสมอ แบ่งเป็น 4 ประเภทหลัก
Corrective Maintenance การตามแก้ไขบั๊กที่พบหลังการใช้งานจริง เช่น การปรับจูน AI Agent เมื่อพบการตอบสนองที่ผิดพลาด
Adaptive Maintenance การปรับระบบให้ทันสมัยตามสภาพแวดล้อมที่เปลี่ยนไป เช่น การอัปเดต Library หรือการปรับโครงสร้างตาม API ของ AI รุ่นใหม่ๆ
Perfective Maintenance การเพิ่มฟีเจอร์ใหม่ตามความต้องการของผู้ใช้ เพื่อให้ระบบสมบูรณ์แบบยิ่งขึ้น
Preventive Maintenance การปรับปรุงเชิงป้องกัน เช่น การทำ Database Optimization เพื่อรองรับปริมาณข้อมูลที่จะเพิ่มขึ้นในอนาคต//

**การเตรียมตัวส่งมอบและคู่มือ (Documentation)**
//เพื่อให้ระบบสามารถส่งต่อและพัฒนาต่อยอดได้ในอนาคต จำเป็นต้องมีเอกสารที่ครอบคลุม 3 ส่วนหลัก
System Documentation รวบรวมพิมพ์เขียวทั้งหมด เช่น แผนภาพ DFD, UML และ Data Dictionary สำหรับนักพัฒนา
User Documentation คู่มือแนะนำการใช้งานสำหรับนักศึกษาและอาจารย์ เพื่อลดภาระในการตอบคำถามการใช้งานเบื้องต้น
Operations Documentation รายละเอียดทางเทคนิคสำหรับการดูแลระบบ เช่น วิธีการ Deploy บน Vercel, การจัดการ Secret Key และขั้นตอนการสำรองข้อมูลใน Supabase//

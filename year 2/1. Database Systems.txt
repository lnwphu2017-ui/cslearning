1. Database Systems
บทที่ 1 สถาปัตยกรรมและโมเดลฐานข้อมูล (The Foundation of Modern Data)

**สถาปัตยกรรม 3 ระดับ (Three-Schema Architecture)** 
//เพื่อให้ระบบฐานข้อมูลมีความยืดหยุ่น นักวิทยาศาสตร์คอมพิวเตอร์จึงออกแบบสถาปัตยกรรมที่แบ่งมุมมองออกเป็น 3 ชั้น ดังนี้
ระดับภายนอก (External Schema / View Level)
เป็นชั้นที่อยู่ใกล้ผู้ใช้ที่สุด คือ "หน้าตาข้อมูล" ที่แต่ละคนเห็น ซึ่งอาจจะต่างกันไปตามสิทธิ์การใช้งาน เช่น ในระบบมหาวิทยาลัย นักศึกษา จะเห็นเฉพาะเกรดของตัวเอง แต่ อาจารย์ จะเห็นรายชื่อและเกรดของนักศึกษาทั้งคลาส
ระดับตรรกะ (Conceptual Schema / Logical Level)
คือการออกแบบโครงสร้าง "ตาราง" ทั้งหมดในระบบ ว่ามี Entity อะไรบ้าง (เช่น Student, Course, Grade) และมีความสัมพันธ์ (Relationship) กันอย่างไร ชั้นนี้จะไม่มีใครสนใจว่าข้อมูลจะถูกเก็บใน Hard Drive ลูกไหน แต่สนใจว่า "ข้อมูลมีโครงสร้างอย่างไร"
ระดับภายใน (Internal Schema / Physical Level)
เป็นชั้นที่ต่ำที่สุดที่คุยกับฮาร์ดแวร์ ว่าข้อมูลจะถูกจัดเก็บลง Disk ในรูปแบบใด มีการทำ Index เพื่อให้ค้นหาเร็วขึ้นอย่างไร หรือมีการบีบอัดข้อมูล (Compression) แบบไหนเพื่อให้ประหยัดพื้นที่และเพิ่ม Performance สูงสุด//

**ความเป็นอิสระของข้อมูล (Data Independence)**
//การแยกชั้นแบบนี้มีประโยชน์มหาศาล เพราะมันทำให้เกิดสิ่งที่เรียกว่า Data Independence ซึ่งแบ่งเป็น 2 รูปแบบ
Logical Data Independence เราสามารถเปลี่ยนโครงสร้างตารางหลังบ้าน (Conceptual Level) ได้ เช่น การแยกตารางใหญ่ๆ ออกเป็นสองตารางเพื่อความสะอาดของข้อมูล โดยที่ไม่ต้องไปนั่งแก้โค้ดแอปพลิเคชันหน้าบ้าน (External Level) ใหม่ทั้งหมด
Physical Data Independence เราสามารถเปลี่ยนฮาร์ดแวร์ หรือเปลี่ยนวิธีการเก็บไฟล์ข้อมูล (Internal Level) เช่น ย้ายจาก HDD ไป SSD หรือเปลี่ยน Format การเก็บไฟล์ เพื่อเพิ่มความเร็ว โดยที่โครงสร้างตาราง (Conceptual Level) ยังคงเหมือนเดิมทุกประการ//

**มุมมองที่กว้างกว่า จาก Relational สู่ AI-Ready Data**
//แม้ว่าเราจะเริ่มบทเรียนด้วยโครงสร้างตารางแบบดั้งเดิม แต่ในโลกของการพัฒนาซอฟต์แวร์ยุคใหม่ (โดยเฉพาะระบบที่มีการผสาน AI) เราไม่ได้มีแค่ตารางแถวและคอลัมน์ครับ//

**NoSQL** //สำหรับข้อมูลที่ไม่มีโครงสร้างชัดเจนและต้องการความเร็วในการสเกลสูง//

**Vector Databases** //สำหรับการเก็บข้อมูลในรูปแบบ "พิกัดทางคณิตศาสตร์" เพื่อให้ AI (เช่น ระบบ RAG) สามารถค้นหาความหมายที่ใกล้เคียงกันได้ แทนที่จะค้นหาแค่คำที่ตรงกันเป๊ะๆ//


บทที่ 2: การออกแบบด้วย ER-Model (Designing with Entity-Relationship Model)

**องค์ประกอบพื้นฐานของ ER-Model**
//ER-Model เป็นแผนภาพเชิงแนวคิด (Conceptual Design) ที่ใช้สัญลักษณ์เรขาคณิตเพื่ออธิบายโครงสร้างข้อมูล โดยมีเสาหลัก 3 ต้นดังนี้:
Entity (เอนทิตี) สิ่งของ บุคคล สถานที่ หรือเหตุการณ์ที่ "มีอยู่จริง" และเราต้องการเก็บข้อมูลของมัน แทนด้วยสัญลักษณ์ สี่เหลี่ยมผืนผ้า
ตัวอย่าง: ลูกค้า (Customer), สินค้า (Product), ใบสั่งซื้อ (Order)
Attribute (แอตทริบิวต์) คุณสมบัติหรือรายละเอียดที่ใช้อธิบาย Entity แทนด้วยสัญลักษณ์ วงรี
ตัวอย่าง: Entity "ลูกค้า" อาจจะมี Attribute คือ รหัสลูกค้า, ชื่อ, เบอร์โทรศัพท์
คีย์หลัก (Primary Key) คือ Attribute ที่มีค่าไม่ซ้ำกันเลย (Unique) ใช้เพื่อระบุตัวตนของ Entity นั้นๆ อย่างชัดเจน (มักขีดเส้นใต้ที่ชื่อ)
Relationship (ความสัมพันธ์) กริยาหรือความเชื่อมโยงระหว่าง Entity แทนด้วยสัญลักษณ์ สี่เหลี่ยมข้าวหลามตัด (Diamond)
ตัวอย่าง: ลูกค้า "ทำการสั่งซื้อ" สินค้า//

**อัตราส่วนความสัมพันธ์ (Cardinality)**
//สิ่งสำคัญที่สุดในการวาด ER-Diagram ไม่ใช่แค่บอกว่าอะไรสัมพันธ์กับอะไร แต่ต้องระบุ "ปริมาณ" ของความสัมพันธ์นั้นด้วย ซึ่งแบ่งออกเป็น 3 รูปแบบหลัก
A. แบบ 1 ต่อ 1 (One-to-One / 1:1)
Entity หนึ่งตัว จะมีความสัมพันธ์กับอีก Entity หนึ่งตัวพอดีเป๊ะ
ตัวอย่าง พนักงาน 1 คน มีบัตรประจำตัวได้ 1 ใบ และบัตร 1 ใบก็เป็นของพนักงานได้แค่ 1 คนเท่านั้น (มักใช้เพื่อแยกข้อมูลที่เป็นความลับหรือข้อมูลที่ไม่ได้เรียกใช้บ่อยๆ ออกจากตารางหลัก)
B. แบบ 1 ต่อกลุ่ม (One-to-Many / 1:N)
Entity หนึ่งตัว สามารถมีความสัมพันธ์กับอีก Entity ได้หลายตัว นี่คือรูปแบบที่พบได้บ่อยที่สุดในฐานข้อมูล
ตัวอย่าง ลูกค้า 1 คน สามารถมี "ใบสั่งซื้อ" ได้หลายใบ แต่ใบสั่งซื้อ 1 ใบ จะต้องเป็นของลูกค้าเพียง 1 คนเท่านั้น//

**แบบกลุ่มต่อกลุ่ม (Many-to-Many / M:N)**
//Entity หลายตัว สามารถมีความสัมพันธ์กับอีก Entity ได้หลายตัวสลับกันไปมา
ตัวอย่าง นักศึกษา 1 คน สามารถลงทะเบียนเรียนได้ "หลายวิชา" และวิชา 1 วิชา ก็สามารถมีนักศึกษาลงเรียนได้ "หลายคน"
ข้อควรระวัง ในฐานข้อมูลเชิงสัมพันธ์ (Relational Database) จริงๆ จะไม่สามารถสร้างความสัมพันธ์แบบ M:N ได้โดยตรง เราจะต้องทำการ "แตก" ความสัมพันธ์นี้ออกโดยใช้ตารางตรงกลาง (Junction Table) ในขั้นตอนต่อไป//

**ประโยชน์ของ Conceptual Design**
//ทำไมเราต้องเสียเวลาวาด ER-Model?
การสื่อสารER-Diagram เป็นภาษากลางที่เข้าใจง่าย ทั้งโปรแกรมเมอร์, นักวิเคราะห์ระบบ (SA), และเจ้าของธุรกิจ (User) สามารถดูแผนภาพนี้และยืนยันความถูกต้องตรงกันได้ก่อนลงมือสร้างจริง
ลดความซ้ำซ้อน การกาง Entity และ Attribute ออกมาให้เห็นทั้งหมด จะช่วยให้เรามองเห็นข้อมูลที่อาจจะซ้ำซ้อนหรือขัดแย้งกัน และจัดการแก้ปัญหาได้ตั้งแต่ในกระดาษ//

บทที่ 3: โมเดลความสัมพันธ์ (Relational Model)

**โครงสร้างของตาราง เมื่อ Attribute กลายเป็นคอลัมน์**
//ในโมเดลความสัมพันธ์ เรามีการเรียกชื่อเฉพาะที่นักพัฒนาและ Database Administrator (DBA) ควรรู้เพื่อให้สื่อสารกันได้อย่างแม่นยำ:
Attribute (คอลัมน์) คือคุณลักษณะของข้อมูล เช่น Student_ID, Name, GPA
Tuple (แถว/เรคคอร์ด) คือข้อมูล 1 ชุดที่เกิดขึ้นจริง เช่น ข้อมูลของนักศึกษาที่ชื่อ "พู" ทั้งแถว
Domain: คือขอบเขตหรือประเภทของข้อมูลที่อนุญาตให้ใส่ใน Attribute นั้นๆ (เช่น Domain ของ Age ต้องเป็นตัวเลขจำนวนเต็มบวกเท่านั้น)//

**กฎเหล็กของการระบุตัวตน คีย์ (Keys)**
//เพื่อให้ฐานข้อมูลทำงานได้อย่างมีประสิทธิภาพและไม่มีข้อมูลที่สับสน เราจำเป็นต้องมี "คีย์" เพื่อควบคุมความถูกต้อง (Integrity):
Primary Key (คีย์หลัก) คือ Attribute (หรือกลุ่มของ Attribute) ที่มีคุณสมบัติ 2 อย่างคือ 1. ต้องไม่ซ้ำ (Unique) และ 2. ห้ามเป็นค่าว่าง (Not Null) เพื่อใช้ระบุตัวตนของแต่ละ Tuple อย่างเจาะจง
Candidate Key คือ Attribute ใดๆ ที่มีความสามารถพอจะเป็นคีย์หลักได้ (แต่เราเลือกมาใช้เพียงตัวเดียวเป็น Primary Key)
Composite Key ในบางกรณี คอลัมน์เดียวอาจไม่พอที่จะระบุความ Unique เราจึงต้องนำ 2 คอลัมน์มารวมกันเพื่อทำหน้าที่เป็นคีย์หลัก//

**การเชื่อมโยงโลกของข้อมูล: คีย์นอก (Foreign Key)**
//นี่คือหัวใจที่ทำให้ฐานข้อมูลนี้เรียกว่า "เชิงสัมพันธ์" ครับ Foreign Key (คีย์นอก) คือ Attribute ในตารางหนึ่งที่ไปอ้างอิงถึง Primary Key ของอีกตารางหนึ่ง
Referential Integrity เป็นกฎที่ระบุว่า ค่าใน Foreign Key จะต้องมีอยู่จริงในตารางที่มันไปอ้างอิงถึงเสมอ
ตัวอย่าง ถ้าเรามีตาราง Enrollment ที่เก็บ Course_ID เป็นคีย์นอก เราจะลงทะเบียนในวิชาที่ไม่มีอยู่จริงในตาราง Course ไม่ได้
ประโยชน์ ป้องกันปัญหา "ข้อมูลกำพร้า" (Orphan Records) ที่อาจเกิดขึ้นเมื่อเราลบข้อมูลตารางหลักแต่ลืมลบตารางที่เกี่ยวข้อง//

**การแปลง ER-Diagram สู่ตาราง (Mapping)**
//กฎเบื้องต้นในการแปลงที่คุณต้องจำไว้คือStrong Entity กลายเป็น 1 ตาราง Attribute กลายเป็น Column
ความสัมพันธ์แบบ 1:N ให้นำ Primary Key ของฝั่ง 1 ไปวางเป็น Foreign Key ในฝั่ง N
ความสัมพันธ์แบบ M:N ต้องสร้างตารางใหม่ขึ้นมา 1 ตาราง (Junction Table) เพื่อเก็บคีย์หลักของทั้งสองฝั่ง//

บทที่ 4: ภาษา SQL สำหรับการจัดการข้อมูล (SQL: The Language of Data)

**DDL (Data Definition Language) การสร้างโครงสร้าง**
//DDL คือคำสั่งที่ใช้จัดการกับ "Schema" หรือโครงสร้างของตาราง เปรียบเสมือนการสร้างบ้านก่อนที่จะย้ายเฟอร์นิเจอร์เข้าไป
CREATE สร้างตารางหรือฐานข้อมูลใหม่
ALTER แก้ไขโครงสร้างตารางที่มีอยู่แล้ว (เช่น เพิ่มคอลัมน์ใหม่)
DROP ลบตารางหรือฐานข้อมูลทิ้ง (คำสั่งนี้ต้องใช้อย่างระมัดระวังที่สุด!)//

**DML (Data Manipulation Language) การจัดการข้อมูล**
//DML คือคำสั่งที่ใช้จัดการกับตัวข้อมูลที่อยู่ในตาราง (Tuples)
INSERT การเพิ่มข้อมูลใหม่ลงในตาราง
UPDATE การแก้ไขข้อมูลเดิมที่มีอยู่
DELETE การลบข้อมูลบางแถวออก
กฎเหล็กของนักพัฒนา อย่าลืมใส่ WHERE clause ในคำสั่ง UPDATE และ DELETE เสมอ มิฉะนั้นคุณอาจจะเปลี่ยนหรือลบข้อมูลทั้งตารางโดยไม่ตั้งใจ!//

**DQL (Data Query Language) ศิลปะการดึงข้อมูล**
//นี่คือหัวใจของ SQL ครับ การดึงข้อมูลไม่ใช่แค่การใช้ SELECT * FROM table แต่คือการกรองและประกอบร่างข้อมูลเพื่อให้ได้สิ่งที่ต้องการ
การประกอบร่างข้อมูลด้วย Joins
ในระบบฐานข้อมูลเชิงสัมพันธ์ เรามักแยกข้อมูลไว้หลายตารางเพื่อลดความซ้ำซ้อน เราจึงต้องใช้ Join เพื่อดึงข้อมูลเหล่านั้นกลับมารวมกัน:
Inner Join ดึงเฉพาะข้อมูลที่มีค่าตรงกันทั้งสองตาราง
Left Join ดึงข้อมูลทั้งหมดจากตารางซ้าย แม้ในตารางขวาจะไม่มีข้อมูลที่ตรงกันก็ตาม
Right Join ดึงข้อมูลทั้งหมดจากตารางขวาเป็นหลัก
Full Outer Join ดึงข้อมูลทั้งหมดจากทุกตารางที่มีความเกี่ยวข้องกัน//

**การสรุปผลด้วย Aggregate Functions**
//ใช้สำหรับการคำนวณข้อมูลเชิงสถิติ เช่น COUNT() (นับจำนวน), SUM() (หาผลรวม), AVG() (ค่าเฉลี่ย) ซึ่งมักใช้คู่กับคำสั่ง GROUP BY เพื่อจัดกลุ่มข้อมูลตามหมวดหมู่//

**Subqueries (คำสั่งซ้อนคำสั่ง)**
//คือการนำผลลัพธ์จากคำสั่ง SELECT หนึ่ง ไปใช้เป็นเงื่อนไขในอีกคำสั่งหนึ่ง ช่วยให้เราเขียน Query ที่มีความซับซ้อนและยืดหยุ่นสูงได้//

**เคล็ดลับเพิ่มประสิทธิภาพ (Performance Tips)**
//ในฐานะที่คุณให้ความสำคัญกับความเร็วของระบบ (Optimization) การเขียน SQL ให้ทำงานได้ดีนั้นมีจุดที่ต้องระวัง
Avoid SELECT * ควรระบุชื่อคอลัมน์ที่ต้องการจริงๆ เพื่อลดภาระการส่งข้อมูลและการใช้หน่วยความจำ
Indexing: การสร้าง Index ในคอลัมน์ที่ถูกค้นหาบ่อยๆ (โดยเฉพาะ Foreign Keys) จะช่วยลดเวลาในการ Query ได้อย่างมหาศาล
Explain Plan ใช้คำสั่ง EXPLAIN เพื่อดูว่าฐานข้อมูลวางแผนจะดึงข้อมูลอย่างไร เพื่อหาจุดที่เป็นคอขวด//

บทที่ 5: การทำ Normalization (ชำแหละตารางเพื่อความสมบูรณ์)

**ปัญหาของตารางที่ไม่ผ่าน Normalization (Anomalies)**
//ก่อนจะเริ่มแก้ เราต้องเห็นปัญหาก่อน:
Insertion Anomaly: เพิ่มข้อมูลไม่ได้เพราะขาดข้อมูลบางส่วน (เช่น เพิ่มรายวิชาใหม่ไม่ได้ถ้ายังไม่มีนักศึกษามาลงทะเบียน เพราะ PK ผูกติดกัน)
Update Anomaly: ต้องตามแก้ข้อมูลชุดเดียวกันในหลายๆ ที่ เสี่ยงต่อความขัดแย้ง
Deletion Anomaly: ลบข้อมูลหนึ่งแล้วทำให้อีกข้อมูลหนึ่งหายไปโดยไม่ตั้งใจ (เช่น ลบชื่อนักศึกษาคนสุดท้ายทิ้ง แล้วทำให้ข้อมูลรายวิชานั้นหายไปจากระบบด้วย)//

**ลำดับขั้นของการทำ Normalization**
//กระบวนการนี้ทำเป็นขั้นบันได โดยปกติในระดับการใช้งานทั่วไป 3NF คือจุดที่เหมาะสมที่สุดครับ
1NF (First Normal Form): ข้อมูลต้องเป็นอะตอม
กฎ: ทุกช่อง (Cell) ต้องมีค่าเพียงค่าเดียว (Atomic Value) และไม่มีกลุ่มข้อมูลที่ซ้ำกัน (No Repeating Groups)
วิธีแก้: หากในหนึ่งช่องมีรายชื่อสินค้าหลายอย่าง ให้แยกออกมาเป็นแถวใหม่ หรือแยกตาราง
2NF (Second Normal Form): กำจัดการขึ้นต่อกันบางส่วน
กฎ: ต้องผ่าน 1NF และทุก Attribute ที่ไม่ใช่คีย์ (Non-key Attributes) ต้องขึ้นอยู่กับ "คีย์หลักทั้งหมด" ไม่ใช่แค่บางส่วน (No Partial Functional Dependency)
วิธีแก้: มักเกิดกับตารางที่มี Composite Key (คีย์ผสม) ถ้ามีข้อมูลที่ขึ้นอยู่กับคีย์เพียงตัวเดียว ให้แยกข้อมูลนั้นออกไปสร้างตารางใหม่
3NF (Third Normal Form): กำจัดการขึ้นต่อกันแบบทอดๆ
กฎ: ต้องผ่าน 2NF และ Attribute ที่ไม่ใช่คีย์ "ต้องไม่ขึ้นต่อกันเอง" (No Transitive Dependency)
ตัวอย่าง: ในตารางพนักงาน มีคอลัมน์ รหัสแผนก และ ชื่อแผนก ซึ่ง ชื่อแผนก ดันไปขึ้นอยู่กับ รหัสแผนก อีกทีหนึ่ง แบบนี้ต้องแยกตารางแผนกออกมาครับ
BCNF (Boyce-Codd Normal Form): ขั้นกว่าของ 3NF
กฎ: เป็นกรณีพิเศษที่เข้มงวดกว่า 3NF โดยเน้นว่าทุกตัวกำหนด (Determinant) ต้องเป็นคีย์คู่แข่ง (Candidate Key) เสมอ มักใช้จัดการกับตารางที่มีคีย์ผสมหลายชุดที่ซ้อนทับกัน//

**เมื่อไหร่ที่เราควร "หยุด" Normalization? (The Wildcard)**
//แม้ว่า Normalization จะช่วยเรื่องความถูกต้องของข้อมูล (Integrity) แต่การแยกตารางมากเกินไปก็มี "ราคาที่ต้องจ่าย" ครับ:
Performance Hit: ยิ่งตารางเยอะ ยิ่งต้องใช้คำสั่ง JOIN มากขึ้น ซึ่งกินทรัพยากร CPU และ Memory สูง
Denormalization: ในระบบที่ต้องการความเร็วในการอ่านสูงมากๆ (Read-heavy) หรือในระบบ Data Warehouse นักออกแบบมักจะยอมให้ข้อมูล "ซ้ำซ้อนบ้าง" เพื่อแลกกับความเร็วในการ Query ที่ไม่ต้อง Join หลายชั้น//

บทที่ 6: การจัดการ Transaction (ธุรกรรมและความน่าเชื่อถือของข้อมูล)
**กฎเหล็ก ACID 4 เสาหลักของความถูกต้อง** //เพื่อให้มั่นใจว่าข้อมูลจะยังคงความถูกต้อง (Integrity) แม้จะมีเหตุการณ์ไม่คาดคิด ฐานข้อมูลระดับ Enterprise ต้องมีคุณสมบัติดังนี้
Atomicity (ความเป็นหน่วยย่อย) ต้องทำทุกคำสั่งในชุดให้จบ ถ้าพังแม้แต่จุดเดียว ระบบต้อง Rollback ย้อนกลับทุกอย่างเหมือนไม่เคยเกิดขึ้น
Consistency (ความสอดคล้อง) ข้อมูลต้องเปลี่ยนจากสถานะที่ถูกต้องหนึ่งไปยังอีกสถานะหนึ่งเสมอตามกฎที่ตั้งไว้
Isolation (ความเป็นเอกเทศ) ในขณะที่ธุรกรรมหนึ่งกำลังรันอยู่ ต้องไม่ถูกรบกวนจากธุรกรรมอื่นที่รันพร้อมกัน
Durability (ความคงทน) เมื่อยืนยันความสำเร็จ (Commit) ข้อมูลต้องถูกบันทึกถาวรลง Disk ต่อให้ไฟดับข้อมูลก็ต้องไม่หาย//

**Concurrency Control (การควบคุมสภาวะพร้อมกัน)** //กลไกจัดการเมื่อมีผู้ใช้งานพยายามอัปเดตข้อมูลแถวเดียวกันในเวลาเดียวกันประกอบด้วย
Locks (การล็อก) แบ่งเป็น Shared Lock (S-Lock) ที่อนุญาตให้ผู้อื่นอ่านได้แต่ห้ามเขียน และ Exclusive Lock (X-Lock) ที่ห้ามผู้อื่นทั้งอ่านและเขียนจนกว่าธุรกรรมจะจบ
Isolation Levels การตั้งค่าความเข้มงวดในการล็อก ซึ่งส่งผลต่อความแม่นยำของข้อมูลและความเร็วของระบบ//

**ภาวะชะงักงัน (Deadlock)** //สภาวะที่ธุรกรรมต่างรอคอยการถือครองทรัพยากรซึ่งกันและกันเป็นวงกลมไปตลอดกาล
สาเหตุ เกิดจาก Transaction A รอทรัพยากรจาก B และ Transaction B ก็รอทรัพยากรจาก A
วิธีแก้ DBMS จะใช้ระบบ Deadlock Detection เพื่อตรวจหาการรอที่เป็นวงกลม แล้วเลือกยกเลิกธุรกรรมหนึ่งทิ้งเพื่อให้ระบบเดินหน้าต่อได้//

**มุมมองนักพัฒนา เมื่อ ACID อาจไม่พอในยุค Modern** //การขยับสู่ระบบที่กระจายตัวอยู่ทั่วโลกทำให้เกิดแนวคิดใหม่ที่ยอมรับข้อจำกัดด้านความเร็ว
BASE (Basically Available, Soft state, Eventual consistency) แนวคิดในฝั่ง NoSQL ที่เน้นการพร้อมใช้งานเป็นหลัก
Eventual Consistency การยอมให้ข้อมูลไม่ตรงกันชั่วคราวแต่จะกลับมาถูกต้องแน่นอนในที่สุด เพื่อแลกกับประสิทธิภาพที่สูงขึ้นในระบบขนาดใหญ่//

บทที่ 7: การบริหารความปลอดภัย (Database Security and Administration)
**DCL (Data Control Language) การควบคุมสิทธิ์การใช้งาน** //ในฐานะนักพัฒนา คุณต้องจำกัดสิทธิ์เพื่อป้องกันความเสียหายต่อข้อมูล โดยใช้ภาษา SQL กลุ่ม DCL จัดการเรื่องนี้ประกอบด้วย
GRANT (การให้สิทธิ์) ใช้เพื่อมอบสิทธิ์ในการทำคำสั่งต่างๆ เช่น SELECT, INSERT, UPDATE ให้กับผู้ใช้ (User) หรือกลุ่มผู้ใช้ (Role)
REVOKE (การดึงสิทธิ์คืน) ใช้เพื่อยกเลิกสิทธิ์ที่เคยให้ไว้ เพื่อป้องกันความเสี่ยงหากสถานะของผู้ใช้นั้นเปลี่ยนไป//

**Role-based Access Control (RBAC) การบริหารจัดการตามบทบาท** //แนวคิดการบริหารจัดการสิทธิ์ผู้ใช้นับพันอย่างมีประสิทธิภาพที่เป็นหัวใจหลักในโปรเจกต์ใหญ่ประกอบด้วย
การสร้าง Role เป็นการกำหนดบทบาทตามตำแหน่งงาน เช่น Manager, Developer หรือ Customer Service
การกำหนดสิทธิ์ให้ Role เป็นการระบุว่าแต่ละบทบาทสามารถเข้าถึงหรือจัดการตารางข้อมูลใดได้บ้าง
การผูก User เข้ากับ Role เมื่อมีผู้ใช้งานใหม่เข้ามาเพียงแค่ระบุบทบาท ระบบจะมอบสิทธิ์ทั้งหมดที่จำเป็นให้โดยอัตโนมัติ//

**แผนสำรองและการกู้คืน (Backup & Recovery)** //กฎของความปลอดภัยคือเราต้องพร้อมเสมอเมื่อระบบล่ม โดยมีกลยุทธ์การสำรองข้อมูล 2 รูปแบบหลักและระบบ Log Files
Full Backup การคัดลอกข้อมูลทั้งหมดที่มี ทำให้กู้คืนได้ง่ายและเร็วที่สุดแต่ใช้เวลาและพื้นที่จัดเก็บมาก
Incremental Backup การสำรองเฉพาะข้อมูลที่เปลี่ยนแปลงนับจากครั้งล่าสุด ช่วยประหยัดพื้นที่และเวลาแต่ขั้นตอนการกู้คืนจะซับซ้อนกว่า
การกู้คืนด้วย Log Files การใช้ระบบ Transaction Log เพื่อทำกระบวนการ Redo (ทำคำสั่งที่ Commit แล้วให้จบ) หรือ Undo (ยกเลิกคำสั่งที่ค้างอยู่) เพื่อให้ข้อมูลกลับมาถูกต้องที่สุด//

**สถาปัตยกรรมความปลอดภัยยุคใหม่** //นักออกแบบฐานข้อมูลระดับมืออาชีพต้องคำนึงถึงความปลอดภัยที่มากกว่าแค่สิทธิ์การเข้าถึงพื้นฐาน
Data Encryption การเข้ารหัสข้อมูล (Encryption at Rest) เพื่อป้องกันการอ่านข้อมูลกรณีที่ฮาร์ดดิสก์ถูกขโมย
Audit Logging การเก็บประวัติการเข้าใช้งานอย่างละเอียดว่าใคร เข้าถึงอะไร เมื่อไหร่ เพื่อใช้ในการตรวจสอบย้อนหลังทางดิจิทัล//

บทที่ 8: การจัดการดัชนี (Index Management & Performance Tuning)
**Table Scan vs. Indexing สารบัญคือหัวใจ** //เปรียบเทียบวิธีการเข้าถึงข้อมูลในฐานข้อมูลเพื่อให้เห็นภาพความแตกต่างของความเร็ว
Table Scan (Full Scan) คือการเปิดดูทีละหน้าตั้งแต่หน้าแรกไปจนกว่าจะเจอข้อมูลที่ต้องการ หากข้อมูลอยู่ลำดับท้ายสุดต้องใช้เวลาในการอ่านนานมาก
Indexing คือการใช้ "สารบัญ" เพื่อดูตำแหน่งของข้อมูลแล้วกระโดดไปยังจุดนั้นทันที ช่วยให้ DBMS ไม่ต้องไล่อ่านข้อมูลทุกแถวใน Disk แต่เลือกอ่านเฉพาะจุดที่จัดระเบียบไว้แล้ว//

**โครงสร้างข้อมูลเบื้องหลัง B-Tree และ Hash** //เพื่อให้ Index ทำงานได้รวดเร็ว ข้อมูลจะถูกจัดเก็บในรูปแบบโครงสร้างพิเศษดังนี้
B-Tree (Balanced Tree) Index เป็นรูปแบบมาตรฐานที่นิยมที่สุด ข้อมูลจะถูกจัดเรียงเป็นชั้นเหมือนกิ่งก้านต้นไม้ที่สมดุล รองรับการค้นหาได้หลากหลายทั้งแบบค่าเจาะจง การหาช่วงข้อมูล (BETWEEN) และการเรียงลำดับ (ORDER BY)
Hash Index ใช้ฟังก์ชัน Hash แปลงค่าข้อมูลเป็นตำแหน่งที่อยู่โดยตรง ค้นหาได้เร็วที่สุดแต่มีข้อจำกัดคือใช้ได้เฉพาะการค้นหาแบบเท่ากับ (=) เท่านั้น ไม่สามารถหาช่วงข้อมูลหรือเรียงลำดับได้//

**ศิลปะแห่งความสมดุล Trade-off ที่ต้องแลก** //การสร้าง Index มีราคาที่ต้องจ่าย จึงไม่ควรสร้างในทุกคอลัมน์เนื่องจากส่งผลกระทบดังนี้
SELECT ทำงานได้เร็วขึ้นมหาศาลเพราะลดจำนวนการอ่านข้อมูลจาก Disk
INSERT / UPDATE / DELETE ทำงานช้าลงเนื่องจากระบบต้องเสียเวลาไปเขียน จัดระเบียบ หรือแก้ไขตำแหน่งข้อมูลในไฟล์ Index ใหม่ให้ถูกต้องเสมอ
Storage ต้องใช้พื้นที่จัดเก็บเพิ่มขึ้นเพราะตัว Index เองก็คือไฟล์ข้อมูลชุดหนึ่งที่ต้องเก็บลง Disk//

**เคล็ดลับการเลือกคอลัมน์เพื่อทำ Index** //นักออกแบบฐานข้อมูลจะเลือกสร้าง Index อย่างชาญฉลาดโดยพิจารณาจากปัจจัยสำคัญ
Primary Keys & Foreign Keys ต้องมี Index เสมอเพื่อความเร็วในการเชื่อมโยงข้อมูล (Join)
คอลัมน์ที่อยู่ใน WHERE clause บ่อยๆ เช่น สถานะ วันที่สร้าง หรืออีเมล เพื่อลดเวลาในการกรองข้อมูล
High Cardinality เลือกคอลัมน์ที่มีค่าหลากหลายและไม่ซ้ำกันเยอะ เช่น รหัสพนักงาน จะมีประสิทธิภาพดีกว่าคอลัมน์ที่มีค่าซ้ำกันมากๆ อย่างเพศ//

บทที่ 9: แนวคิดฐานข้อมูล NoSQL (ขยายขีดจำกัดสู่ยุค Big Data)

**การขยายสเกล Vertical vs. Horizontal** //เหตุผลหลักที่ NoSQL ถูกนำมาใช้คือความสามารถในการ Scaling เพื่อรองรับผู้ใช้งานจำนวนมาก
Vertical Scaling (SQL) เปรียบเหมือนการอัปเกรดคอมพิวเตอร์เครื่องเดิมให้แรงขึ้น ซึ่งมีขีดจำกัดด้านฮาร์ดแวร์และมีราคาแพง
Horizontal Scaling (NoSQL) เปรียบเหมือนการเพิ่มคอมพิวเตอร์ราคาถูกหลายเครื่องมาช่วยกันทำงานในระบบกระจาย (Distributed System) ทำให้ขยายระบบได้แทบไม่จำกัด//
Piscine

**ประเภทของ NoSQL และการเลือกใช้งาน** //เราแบ่ง NoSQL ออกเป็น 4 ตระกูลหลักตามรูปแบบการจัดเก็บข้อมูลดังนี้
Document Database (เช่น MongoDB) เก็บข้อมูลลักษณะ JSON/BSON แบบ Schema-less เหมาะกับโปรไฟล์ผู้ใช้หรือระบบจัดการเนื้อหา (CMS) ที่ต้องการความคล่องตัว
Key-Value Store (เช่น Redis) เก็บข้อมูลเป็นคู่กุญแจ-ค่า ทำงานบน Memory ทำให้เร็วระดับแสง เหมาะกับระบบ Session หรือการทำ Caching
Column-Family Store (เช่น Cassandra) เก็บข้อมูลแยกตามคอลัมน์แทนแถว ออกแบบมาเพื่อเขียนข้อมูลมหาศาลพร้อมกัน เหมาะกับงาน Analytics หรือเก็บ Log จำนวนมาก
Graph Database (เช่น Neo4j) เก็บข้อมูลเป็น Node และ Edge เพื่อหาความสัมพันธ์ที่ซับซ้อน เหมาะกับระบบแนะนำเพื่อนหรือการตรวจจับการฉ้อโกง//

**ทฤษฎี CAP กฎการเลือกข้าง** //ในระบบกระจายตัวเราไม่สามารถได้ทุกอย่างพร้อมกัน ตามกฎ CAP Theorem ที่ระบุว่าเราเลือกได้เพียง 2 จาก 3 อย่างเท่านั้น
Consistency (C) ข้อมูลทุกเครื่องในระบบต้องตรงกันทันที
Availability (A) ระบบต้องสามารถตอบสนองและใช้งานได้ตลอดเวลาแม้บางเครื่องจะพัง
Partition Tolerance (P) ระบบยังคงทำงานได้แม้การเชื่อมต่อระหว่างเครือข่ายจะขาดหายไปบางส่วน//

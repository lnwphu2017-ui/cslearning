ชั้นปีที่ 3: วิชาชีพขั้นสูงและการประยุกต์ใช้

1. Software Engineering
Course: Software Engineering
Lesson: บทที่ 1: นิยามวิศวกรรมซอฟต์แวร์
Keywords: Software Engineering, Professionalism, SDLC, Software Crisis, Engineering Principles
Content: วิศวกรรมซอฟต์แวร์ (Software Engineering) ไม่ใช่แค่การเขียนโค้ดให้โปรแกรมทำงานได้ แต่คือ "ศาสตร์แห่งการผลิตซอฟต์แวร์อย่างเป็นระบบ" บทเรียนนี้จะปูพื้นฐานความเข้าใจว่าทำไมเราถึงต้องนำหลักการทางวิศวกรรมมาใช้ เพื่อแก้ปัญหา "วิกฤตการณ์ซอฟต์แวร์" (Software Crisis) ที่โปรแกรมมักจะส่งมอบล่าช้า งบประมาณบานปลาย หรือมีข้อผิดพลาดร้ายแรง ผู้เรียนจะได้เรียนรู้วงจรชีวิตของการพัฒนาซอฟต์แวร์ (SDLC) และการยกระดับตัวเองสู่การเป็น "วิศวกรซอฟต์แวร์" (Software Engineer) ที่ต้องคำนึงถึงความเป็นมืออาชีพ ความคุ้มค่าในการลงทุน (ROI) และการสร้างระบบที่สามารถดูแลรักษาและขยายขนาดได้ในระยะยาว
Course: Software Engineering
Lesson: บทที่ 2: Agile & Scrum Framework
Keywords: Agile Methodology, Scrum Framework, Sprints, Product Owner, Scrum Master, Iterative Development
Content: ในยุคที่ความต้องการของลูกค้าเปลี่ยนแปลงอย่างรวดเร็ว การพัฒนาซอฟต์แวร์แบบดั้งเดิม (Waterfall) ที่ต้องรอให้เสร็จทั้งหมดทีเดียวอาจไม่ตอบโจทย์ บทเรียนนี้จะพาไปรู้จักกับแนวคิด "Agile" ที่เน้นความยืดหยุ่น การทำงานร่วมกัน และการส่งมอบซอฟต์แวร์ที่ใช้งานได้จริงอย่างต่อเนื่อง (Iterative) โดยเจาะลึกกรอบการทำงาน "Scrum Framework" ซึ่งเป็นที่นิยมสูงสุดในอุตสาหกรรมเทคโนโลยี ผู้เรียนจะได้เรียนรู้บทบาทในทีม เช่น Product Owner, Scrum Master และ Development Team รวมถึงกระบวนการทำงานเป็นรอบสั้นๆ ที่เรียกว่า "Sprint" เพื่อให้ทีมสามารถรับข้อเสนอแนะและปรับเปลี่ยนแผนงานได้ทันท่วงที
Course: Software Engineering
Lesson: บทที่ 3: Software Design Hierarchy
Keywords: Software Architecture, System Design, Microservices, Monolithic, UML, Design Principles
Content: ก่อนที่วิศวกรจะสร้างตึก ย่อมต้องมีแบบแปลน การสร้างซอฟต์แวร์ก็เช่นกัน บทเรียนนี้จะสอนการออกแบบสถาปัตยกรรมซอฟต์แวร์ (Software Architecture) ซึ่งเป็นโครงร่างระดับสูงสุดของระบบ ผู้เรียนจะได้เห็นภาพความแตกต่างระหว่างสถาปัตยกรรมแบบรวมศูนย์ดั้งเดิม (Monolithic) กับแบบสมัยใหม่ (Microservices) ที่แยกบริการออกเป็นส่วนย่อยๆ เพื่อให้ง่ายต่อการขยายสเกลระบบ นอกจากนี้ยังครอบคลุมการใช้ภาษาภาพอย่าง UML (Unified Modeling Language) เพื่อสื่อสารการออกแบบกับคนในทีม และหลักการออกแบบระดับโค้ด เพื่อให้สถาปัตยกรรมที่วางไว้ถูกนำไปสร้างจริง (Implement) ได้อย่างแข็งแกร่ง
Course: Software Engineering
Lesson: บทที่ 4: Quality Control (QA)
Keywords: Quality Assurance, Quality Control, Software Standards, Code Review, ISO/IEC, Metrics
Content: ซอฟต์แวร์ที่ดีต้องไม่ใช่แค่ "รันผ่าน" แต่ต้องได้ "คุณภาพ" ตามมาตรฐาน บทเรียนนี้จะแยกความแตกต่างระหว่าง Quality Assurance (QA) ที่เน้นการออกแบบกระบวนการทำงานที่รัดกุมเพื่อป้องกันข้อผิดพลาดตั้งแต่ต้นทาง และ Quality Control (QC) ที่เน้นการตรวจสอบเพื่อหาข้อบกพร่องในตัวผลิตภัณฑ์ ผู้เรียนจะได้เรียนรู้เกี่ยวกับมาตรฐานซอฟต์แวร์ระดับสากล การกำหนดตัวชี้วัดคุณภาพ (Software Metrics) และแนวปฏิบัติในการตรวจสอบโค้ดร่วมกัน (Code Review) เพื่อให้แน่ใจว่าซอฟต์แวร์ที่จะส่งมอบนั้นมีความน่าเชื่อถือ ปลอดภัย และตรงตามข้อกำหนดความต้องการ (Requirements) ของผู้ใช้งาน
Course: Software Engineering
Lesson: บทที่ 5: Comprehensive Testing
Keywords: Software Testing, Unit Testing, Integration Testing, System Testing, User Acceptance Testing (UAT), Test Cases
Content: การทดสอบคือหัวใจสำคัญของการป้องกันหายนะก่อนที่ซอฟต์แวร์จะหลุดไปถึงมือผู้ใช้งาน บทเรียนนี้จะเจาะลึกกระบวนการทดสอบซอฟต์แวร์ในทุกมิติ เริ่มตั้งแต่ระดับฐานรากคือ Unit Testing เพื่อทดสอบฟังก์ชันย่อยทีละส่วน, Integration Testing เพื่อดูความราบรื่นในการเชื่อมต่อระหว่างโมดูล, System Testing สำหรับประเมินภาพรวมทั้งระบบ ไปจนถึงด่านสุดท้ายคือ User Acceptance Testing (UAT) ที่ให้ผู้ใช้งานจริงมาทดสอบเพื่ออนุมัติรับมอบงาน รวมถึงการเรียนรู้วิธีการเขียน Test Cases ที่ครอบคลุมทุกสถานการณ์ (Edge Cases) เพื่อดักจับบั๊กที่อาจซ่อนอยู่ได้อย่างหมดจด
Course: Software Engineering
Lesson: บทที่ 6: Project Management Tools
Keywords: Project Management, Jira, Kanban Boards, Task Tracking, Backlog, Issue Tracking
Content: วิศวกรรมซอฟต์แวร์เป็นงานที่ต้องอาศัยการทำงานเป็นทีม การจัดการโครงการที่มีประสิทธิภาพจึงขาดไม่ได้ บทเรียนนี้จะแนะนำให้รู้จักกับเครื่องมือจัดการโปรเจกต์ระดับองค์กร โดยเน้นที่การใช้งาน "Jira" ซึ่งเป็นซอฟต์แวร์มาตรฐานสำหรับทีม Agile ผู้เรียนจะได้เรียนรู้วิธีการสร้างและบริหารจัดการ Product Backlog (รายการสิ่งที่ต้องทำ), การประเมินเวลาและความยากของงาน (Story Points), และการใช้งานกระดาน Kanban เพื่อติดตามสถานะของงานแบบเรียลไทม์ ตั้งแต่เริ่ม (To Do) กำลังทำ (In Progress) ไปจนถึงเสร็จสิ้น (Done) เพื่อให้ทุกคนในทีมเห็นเป้าหมายและขับเคลื่อนโปรเจกต์ไปพร้อมกัน
Course: Software Engineering
Lesson: บทที่ 7: Version Control (Git)
Keywords: Version Control, Git, GitHub, GitLab, Branching, Merging, Pull Requests, Conflict Resolution
Content: เมื่อโปรแกรมเมอร์หลายคนต้องแก้โค้ดในโปรเจกต์เดียวกัน หากไม่มีระบบจัดการที่ดี โค้ดจะทับซ้อนและพังทลายอย่างแน่นอน "ระบบควบคุมเวอร์ชัน" (Version Control System) คือทางออก บทเรียนนี้จะเจาะลึกการใช้งาน "Git" เครื่องมือมาตรฐานโลกที่ช่วยบันทึกประวัติการเปลี่ยนแปลงของโค้ดทุกบรรทัด ผู้เรียนจะได้เรียนรู้กลยุทธ์การแตกสาขาโค้ด (Branching) เพื่อแยกกันทำงานฟีเจอร์ใหม่ การนำโค้ดมารวมกัน (Merging), การขออนุมัติรวมโค้ดผ่าน Pull Requests, และเทคนิคการแก้ปัญหาเมื่อเกิดการแก้ไขโค้ดชนกัน (Conflict Resolution) ผ่านแพลตฟอร์มอย่าง GitHub หรือ GitLab
Course: Software Engineering
Lesson: บทที่ 8: ระบบนิเวศการส่งมอบซอฟต์แวร์
Keywords: Software Maintenance, DevOps, Deployment, Incident Management, Legacy Systems, Post-Release Support
Content: งานของวิศวกรซอฟต์แวร์เพิ่งจะเริ่มต้นขึ้นจริงๆ ในวันที่ระบบเปิดใช้งาน (Go-Live) บทเรียนสุดท้ายนี้จะอธิบายถึงช่วงชีวิตที่ยาวนานที่สุดของซอฟต์แวร์ นั่นคือ "การบำรุงรักษาและการสนับสนุน" (Maintenance & Support) ผู้เรียนจะได้เข้าใจวงจรการรับมือและแก้ไขบั๊กที่ผู้ใช้รีพอร์ตเข้ามา (Incident Management), การอัปเดตระบบให้เข้ากับเทคโนโลยีใหม่ๆ, และการรับมือกับระบบเก่า (Legacy Systems) นอกจากนี้ยังจะปูพื้นฐานแนวคิด DevOps ที่ผสานการทำงานของทีมพัฒนา (Dev) และทีมปฏิบัติการ (Ops) เข้าด้วยกัน เพื่อให้การส่งมอบซอฟต์แวร์เวอร์ชันใหม่ๆ เป็นไปอย่างอัตโนมัติ รวดเร็ว และปลอดภัยสูงสุด

2. UX/UI Design
Course: UX/UI Design
Lesson: บทที่ 1: พื้นฐานทางศิลปะและจิตวิทยา
Keywords: Color Theory, Typography, Visual Hierarchy, Psychology of Design, Gestalt Principles
Content: การออกแบบ UI ที่ดีไม่ได้อาศัยแค่สัญชาตญาณ แต่มีรากฐานมาจากศิลปะและจิตวิทยา บทเรียนนี้จะพาผู้เรียนไปทำความเข้าใจ "จิตวิทยาการออกแบบ" (Psychology of Design) เช่น ทฤษฎีสี (Color Theory) ที่สีแต่ละเฉดสามารถกระตุ้นอารมณ์และพฤติกรรมของผู้ใช้ที่แตกต่างกัน, การจัดความสำคัญทางสายตา (Visual Hierarchy) ผ่านขนาดของตัวอักษร (Typography) และพื้นที่ว่าง (White Space) เพื่อนำสายตาผู้ใช้ไปยังจุดที่สำคัญที่สุด รวมถึงทฤษฎีเกสตัลต์ (Gestalt Principles) ที่อธิบายว่าสมองของมนุษย์รับรู้และจัดกลุ่มภาพที่เห็นอย่างไร ความรู้เหล่านี้คืออาวุธลับที่จะทำให้หน้าแอปพลิเคชันของคุณดูเป็นมืออาชีพและใช้งานง่ายตั้งแต่แรกเห็น
Course: UX/UI Design
Lesson: บทที่ 2: Research and Persona
Keywords: User Research, User Persona, Empathy Map, User Journey, Target Audience
Content: กฎเหล็กของ UX คือ "คุณไม่ใช่ผู้ใช้งาน" (You are not the user) ดังนั้นก่อนที่จะเริ่มออกแบบหน้าจอ เราต้องเข้าใจปัญหาที่แท้จริงเสียก่อน บทเรียนนี้จะสอนกระบวนการ "วิจัยผู้ใช้" (User Research) ด้วยวิธีการต่างๆ เช่น การสัมภาษณ์แบบลึก หรือการทำแบบสอบถาม จากนั้นนำข้อมูลที่ได้มาสกัดเป็น "User Persona" หรือตัวละครสมมติที่เป็นตัวแทนของกลุ่มเป้าหมายหลัก ควบคู่ไปกับการทำแผนที่ความรู้สึก (Empathy Map) เพื่อให้ทีมพัฒนาเข้าใจถึงความต้องการ ความคับข้องใจ (Pain Points) และพฤติกรรมของลูกค้าอย่างแท้จริง การเริ่มต้นด้วยความเข้าใจที่ถูกต้อง จะช่วยให้เราสร้างผลิตภัณฑ์ที่คนอยากใช้และแก้ปัญหาได้ตรงจุด
Course: UX/UI Design
Lesson: บทที่ 3: Visual Information Design
Keywords: Information Architecture, Data Visualization, Wireframing, Navigation, Cognitive Load
Content: เมื่อเรามีข้อมูลหรือเนื้อหาจำนวนมหาศาลบนแอปพลิเคชัน ความท้าทายคือจะจัดวางอย่างไรไม่ให้ผู้ใช้รู้สึกสับสนและเลิกใช้งาน บทเรียนนี้มุ่งเน้นที่ "สถาปัตยกรรมข้อมูล" (Information Architecture - IA) ซึ่งเป็นการจัดระเบียบโครงสร้างของเว็บไซต์และระบบนำทาง (Navigation) ให้เป็นตรรกะ ผู้เรียนจะได้เรียนรู้วิธีการลด "ภาระทางปัญญา" (Cognitive Load) หรือความเหนื่อยล้าของสมองในการประมวลผลข้อมูล ผ่านการทำ Data Visualization เพื่อเปลี่ยนตัวเลขให้เป็นภาพที่เข้าใจง่าย และการสร้างโครงร่างหน้าจอเบื้องต้น (Wireframing) เพื่อวางตำแหน่งของเนื้อหาและปุ่มต่างๆ ก่อนที่จะลงมือใส่สีสันและรายละเอียด
Course: UX/UI Design
Lesson: บทที่ 4: Prototyping Skills
Keywords: Prototyping, Figma, High-Fidelity, Mockup, User Flow, Interactive Design
Content: จากภาพร่างบนกระดาษ สู่หน้าจอที่คลิกได้จริง บทเรียนนี้คือภาคปฏิบัติในการแปลงไอเดียให้กลายเป็น "แบบจำลองเสมือนจริง" (High-Fidelity Prototype) ผู้เรียนจะได้ทำความรู้จักและฝึกใช้งานเครื่องมือมาตรฐานอุตสาหกรรมอย่าง Figma (หรือเครื่องมืออื่นๆ ที่ใกล้เคียง) เพื่อสร้าง Mockup หน้าจอที่มีความสวยงามสมบูรณ์แบบ ควบคู่ไปกับการกำหนด "เส้นทางการใช้งาน" (User Flow) เพื่อจำลองว่าเมื่อผู้ใช้กดปุ่มนี้ หน้าจอจะเปลี่ยนไปอย่างไร การทำ Prototype ที่ดีจะช่วยให้นักพัฒนาเห็นภาพตรงกันกับนักออกแบบ และสามารถนำไปใช้เสนองานให้กับลูกค้าหรือนักลงทุนได้อย่างมืออาชีพ
Course: UX/UI Design
Lesson: บทที่ 5: Micro-interactions
Keywords: Micro-interactions, Animation, Feedback, User Engagement, Delight, State Changes
Content: ความแตกต่างระหว่างแอปพลิเคชันที่ใช้งานได้ กับแอปพลิเคชันที่ใช้งานแล้วรู้สึก "ว้าว" มักอยู่ที่รายละเอียดเล็กๆ ที่เรียกว่า "Micro-interactions" บทเรียนนี้จะเจาะลึกการสร้างลูกเล่นและการตอบสนองระดับย่อย เช่น แอนิเมชันปุ่มกดที่เปลี่ยนสีเมื่อนำเมาส์ไปชี้ (Hover State), ไอคอนหัวใจที่เด้งขึ้นเมื่อกดไลก์, หรือแถบ Loading Screen ที่ดูมีชีวิตชีวา หน้าที่หลักของ Micro-interactions ไม่ใช่แค่ความสวยงาม แต่คือการ "ให้ฟีดแบ็ก" (Feedback) แก่ผู้ใช้ว่าระบบได้รับคำสั่งแล้ว และกำลังดำเนินการอยู่ การใส่ใจในรายละเอียดเหล่านี้จะช่วยเพิ่มความพึงพอใจ (Delight) และทำให้ประสบการณ์การใช้งานเป็นไปอย่างราบรื่น

Content: ในอดีต หากบริษัทต้องการสร้างเว็บไซต์ พวกเขาต้องลงทุนซื้อเครื่องเซิร์ฟเวอร์ราคาแพงมาตั้งไว้ในห้องแอร์เย็นเฉียบ แต่แนวคิด "คลาวด์คอมพิวติ้ง" (Cloud Computing) ได้เข้ามาพลิกโฉมโลกใบนี้ โดยเปลี่ยนจากการเป็น "เจ้าของฮาร์ดแวร์" มาเป็นการ "เช่าใช้ทรัพยากร" ผ่านอินเทอร์เน็ต บทเรียนนี้จะปูพื้นฐานคุณลักษณะสำคัญของคลาวด์ เช่น การแชร์ทรัพยากรร่วมกันระดับโลก (Resource Pooling) และการเรียกใช้งานได้ทันทีเมื่อต้องการ (On-demand Self-service) นอกจากนี้ ผู้เรียนจะได้ทำความเข้าใจโมเดลบริการทั้ง 3 ระดับ ได้แก่ IaaS (เช่าโครงสร้างพื้นฐาน เช่น เครื่องเซิร์ฟเวอร์เปล่า), PaaS (เช่าแพลตฟอร์มที่พร้อมให้โปรแกรมเมอร์เขียนโค้ดลงไปได้เลย), และ SaaS (ซอฟต์แวร์สำเร็จรูปที่พร้อมใช้งาน เช่น Gmail หรือ Google Drive) การเข้าใจคอนเซปต์เหล่านี้คือจุดเริ่มต้นของการออกแบบระบบที่คุ้มค่าและยืดหยุ่น

Keywords: Disaster Recovery (DR), Backup, RTO, RPO, Snapshots, Multi-Region Deployment
Content: แม้คลาวด์จะมีเสถียรภาพสูง แต่ภัยพิบัติ โค้ดที่ผิดพลาด หรือการถูกโจมตีทางไซเบอร์ย่อมเกิดขึ้นได้เสมอ วิศวกรระบบที่ดีจึงต้องมีแผน "การกู้คืนเหตุฉุกเฉิน" (Disaster Recovery - DR) บทเรียนนี้มุ่งเน้นกลยุทธ์การสำรองข้อมูล (Backup Strategies) อย่างปลอดภัย ผู้เรียนจะได้รู้จักกับเครื่องมืออย่าง "Snapshots" ที่สามารถบันทึกสถานะของเซิร์ฟเวอร์หรือฐานข้อมูลทั้งก้อนไว้ ณ ช่วงเวลาหนึ่ง และสามารถย้อนคืนกลับมาได้ในไม่กี่นาที รวมไปถึงการคำนวณเป้าหมายความเสียหายที่ยอมรับได้ผ่านค่า RPO (ข้อมูลหายได้มากสุดกี่ชั่วโมง) และ RTO (ระบบล่มได้นานสุดกี่ชั่วโมงก่อนธุรกิจจะเสียหาย) ตลอดจนการออกแบบสถาปัตยกรรมแบบ Multi-Region ที่ก๊อปปี้ระบบไปตั้งไว้อีกทวีปหนึ่ง เพื่อให้ระบบทำงานต่อได้ทันทีแม้ศูนย์ข้อมูลหลักจะเกิดไฟไหม้ก็ตาม

Content: หนึ่งในหลุมพรางที่น่ากลัวที่สุดสำหรับผู้เริ่มต้นใช้คลาวด์คือ "บิลค่าใช้จ่ายตอนสิ้นเดือนที่พุ่งสูงปรี๊ด" เนื่องจากความง่ายในการกดสร้างเซิร์ฟเวอร์ บทเรียนสุดท้ายนี้จะเจาะลึกศาสตร์แห่ง "FinOps" (Financial Operations) หรือการบริหารงบประมาณบนคลาวด์ ผู้เรียนจะได้เรียนรู้วิธีการคำนวณต้นทุนรวม (TCO) ก่อนย้ายระบบขึ้นคลาวด์ เทคนิคการเลือกใช้อินสแตนซ์ให้คุ้มค่า เช่น การจองล่วงหน้าระยะยาว (Reserved Instances) เพื่อรับส่วนลดสูงถึง 70% หรือการประมูลทรัพยากรว่าง (Spot Instances) สำหรับงานประมวลผลที่ไม่ซีเรียสเรื่องการถูกขัดจังหวะ ตลอดจนการตั้งระบบแจ้งเตือน (Billing Alarms) และแดชบอร์ดวิเคราะห์ค่าใช้จ่าย เพื่อให้แน่ใจว่าทุกบาทที่องค์กรจ่ายไปนั้นเกิดความคุ้มค่าและไม่รั่วไหล
